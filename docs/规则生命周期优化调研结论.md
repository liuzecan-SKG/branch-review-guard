> 方法：按对抗式编排速查卡「开放式生成」剧本执行——共享事实包 → 4 个互斥取向设计代理并行独立成案（A 最小改动收敛 / B 审核体验 / C 自动化节流 / D 度量驱动）→ 每案 1 个怀疑者对承重论断做 file 级质证 → 裁判按预公布 rubric（痛点解决 30 / 纪律红线 25 / 成本可移植 20 / 度量协同 15 / 爆炸半径 10）打分并嫁接。9 个子代理，无失败。

## 零、压过一切方案的实测发现：本地规则加载不可靠（P0）

四案共同地基是"branch-review-rules/ 放进即生效"，但怀疑者从历史报告里挖出**两层实测失灵**：

- **子目录层**：`branch-review-guard-branch-a7c5d7e0c-20260713-thorough.md:5` 写"项目本地 branch-review-rules/ 仅空目录，无叠加"——而当天该目录 `discover-new/` 子目录里已有 14 条规则（7 月 6 日落位）。即这 14 条曾整批静默失效。
- **根目录层**：`branch-review-guard-diff-a7c5d7e0c-20260714.md` 晚于根目录 3 条 calibration 创建约 4 小时，仍未加载——被 `calibration-local-pom-isolation` 豁免的 pom 问题被列为 Top-1 P1。

根因：SKILL.md:33 只承诺读"项目根 branch-review-rules/"（无递归字样），且 `consolidate-report.md:30` 的申报是条件式（"若叠加了"）——**加载失败时报告恰好什么都不说**，静默丢失不可见。这意味着"担心审得不准"之外还有更隐蔽的问题：**审对了也可能没生效**。

【推测/待验证】20260714 那次根目录失灵是否出自异构复审的非 Claude 模型，归因未定，修法优先级受其影响。

## 一、用户四个问题的收口答案

### a) 规则收敛到哪、来源职责怎么划

规则只有**两个家**，其余三处明确为**非规则源**：

| 位置 | 职责 |
| --- | --- |
| 项目根 `branch-review-rules/`（个人家） | **唯一首落位**。扁平放根目录、不建子目录；`pack: local`、`id: local/<短名>`；放入即生效、移出即撤销（`enabled` 在此是死字段，SKILL.md:33 明文不走 enabled 判定）。gitignored 纯本地 |
| 插件仓库 `rules/`（团队家） | baseline / skg-spring 作者预置 + `discover-new/` 只收本地服役有战绩后**晋升**的规则。**空是常态不是欠账** |
| 用户级 CLAUDE.md | 写码**预防态**简版红线，允许与 rules 语义重叠（消费时机不同：写码 vs 评审）；今后新增检测类条目**只进 rules**——这是对人的约定不是机制（宿主照常注入，靠去重缓解） |
| 项目 CLAUDE.md | 项目构建/架构事实，不放评审规则 |
| memory | 决策存档与偏好；已规则化内容迁往 rules 或删 |

已发生的分叉照此消解：skg 本地 `branch-review-rules/discover-new/` 14 条一次性移到根、统一 pack/id，删空子目录——本地"discover-new"命名从此消失，插件仓库 `rules/discover-new/` 是唯一的 discover-new。

### b) 审核后落哪、审核怎么变便宜

**两段式落位**（文档向实践投降）：确认后一律先落个人家（只影响自己、分钟级回滚）；本地服役**命中≥3 且存活率≥2/3** 才晋升团队家（四步缺一不生效：改回 id/pack 为 discover-new → commit 插件仓库 → config.yaml 置 enabled → 发版）。样本不足不晋升，允许"我确信"人工 override。

审核便宜化：distill/rule 收尾直接输出 **triage 一页清单**（一行一条：summary｜类型维度｜证据实例数｜建议裁决｜风险），建议裁决必须附**误杀模拟的 file:line 反例**（无反例的 adopt 建议无效，防橡皮图章）；schema/去重/过拟合红线/豁免边界等机械判断由 agent 预审代劳，**人只答一个价值判断："值不值得进我的试用区"**，批量一行裁决，agent 代劳落位/归档/记 LEDGER 台账。

### c) distill 多久跑、怎么触发

**按量不按时**：每积累 5 份新 `branch-review-guard-*` 报告跑一次。机制 = 水位线 + 收尾盘点：distill 跑完写 `rule-drafts/.distill-state`（时间 + 已消化最新报告名；**勿用 mtime 当代理量**——已被证伪会被 /rule 手动草稿污染）；review/diff 收尾盘点，新报告≥5 份**或本次对抗验证否决≥3 条**（误报堆积信号）即在报告尾提示跑 distill。默认只提醒；Claude Code 可选自动跑到"草稿产出"为止——不越人工关卡红线（distill 只写 enabled:false 草稿，绝不写 rules/）。非 Claude Code 形态降级为报告内提醒。

**立即动作**：补跑一次 distill，消化 7 月 9 日后的 8 份增量报告（27 份总数含非输入源前缀）。

### d) "担心审不准"怎么解

不逼人一次审准，三层保险丝把审错代价压到分钟级：

1. **结构限赌注**：首落位只在个人本地，撤销 = 移出一个文件——前提是先修第零节的加载可见性，否则担忧换个更隐蔽的形态。
2. **预审代劳**：最容易审不准的机械判断全交 agent 预审并给带反例的建议裁决，人的判断面从"这条永远对吗"缩到"值不值得试用"。
3. **终审交给数据**（定位调研甲+乙路线最轻版）：报告每条发现标「触发规则: <pack>/<id>」、skeptic 否决标「规则误报: <id>」；distill 顺带回写战绩到 `branch-review-reports/FEEDBACK.md`（**落报告侧、绝不落规则目录**——非规则文件混入加载源有 README 被误计成第 15 条规则的实证前科）；战绩=命中÷存活、命中机会<3 标"样本不足"；calibration 每期附"被压掉的发现清单"反审计棘轮漂移。审错一条，2-3 次评审内被战绩揪出。

## 四、反对意见留存（翻盘条件）

1. **团队规模**：当前单人单项目。一旦规则要多人共用，"本地首落位+晋升"会造成各成员试用区分叉、团队拿不到沉淀——届时主落位应**反转为插件仓库**，triage 命令和战绩门槛从可选变必需。反之若确认长期单人，FEEDBACK.md 可再降档为口头盘点，:triage 永远不建。
2. **加载可靠性**：若无条件申报上线后发现失灵频率仍高（LLM best-effort 读取本质无保证），应反向收口——规则全走插件仓库发版分发、config.yaml 开关管试用，本地目录降级为纯草稿区。这会让整个两段式落位模型作废。
3. **hooks/cron 通道确认可靠**：distill 触发应从 nudge 升级为自动跑到草稿为止（C 的 run 档转正）。

## 五、完整性缺口（批评家核查，未被任何方案覆盖或未验证）

- 插件 cache 分发时滞：所有 prompt/skill 改动必须发版才生效，四案 effort 普遍因此低估（仅 B 显式列了发版）。
- 三处未实测：水位线提醒"够不够用"需运行观察；skeptic 预审是否系统性偏 adopt 需 2-3 次真实 distill 观察；20260714 根目录加载失灵的归因（是否异构复审非 Claude 模型所致）。
- memory 里已规则化内容的迁移/删除盘点，只有原则没有动作清单。
