# 需求到上线开发 SOP — 从一句话需求到可发布代码

> 面向用 branch-review-guard 的开发者。这是**使用实践**，不改插件。它是本仓库评审侧 SOP（`docs/2+2+1合裁评审-SOP.md` / `docs/异构复审-SOP.md`）的**开发侧配对**：评审 SOP 管"代码写完之后怎么验"，本文管"需求来了之后怎么落到代码"。两者末端咬合——本文第四步直接接评审 SOP。

## 一句话

**按需求规模分层走流程：小改直接写，中等补验收标准，大需求走"澄清→PRD→设计擂台→写码→评审"全链路；第一步永远是钉死需求而非生成方案，agent 在需求层是结构化器和质询者，不是产出方。**

## 流水线全貌

```
需求(模糊)
  │
  ▼  第一步：需求澄清 + PRD（人主位，agent 结构化/质询）
 PRD 草稿
  │
  ▼  第二步：方案设计（agent 主位，多视角擂台）—— 仅大需求
 设计文档（精炼稿移入 docs/）
  │
  ▼  第三步：代码实现（agent 主位，按设计文档）
 代码
  │
  ▼  第四步：分支评审 → 接 docs/2+2+1合裁评审-SOP.md
 可发布
```

**闭环点**：第二步产出的精炼设计稿移入 `docs/`，第四步 `/branch-review-guard:review` 建立上下文时**自动读取**——设计文档不白做，前后环节咬合。这也是设计阶段值得投入的回报。

## 准入（先判规模，别默认全套）

| 需求规模 | 链路 | 第一步 |
|---|---|---|
| 一两行能说清的小改 / bugfix | 直接写码 → `:diff` 评审 | 不用 PRD |
| 单个功能点（中等） | 简版 PRD → 写码 → `:diff` | 澄清 + 验收标准 |
| 跨模块 / 新子系统 / 有架构决策 / 动共享代码 | **完整 PRD → `:design` 擂台 → 写码 → `:review`** | 下面的 prompt |

**判断标准跟评审三档同构：这个需求做错了会怎样。** 自己能兜住就轻；返工贵就中等；动核心链路/共享代码/架构决策就走全套。**别过度流程化**——小需求上全套是浪费，违背"最好的代码是不用写的代码"。

## 会话边界（哪几步同会话、哪几步必须拆）

判别式（与评审 SOP"会话隔离=立场独立"同一条地基）：**上一步的推理过程会污染下一步的判断吗？** 会 → 新会话；下一步正需要上一步的上下文 → 同会话。

| 环节 | 会话安排 | 原因 |
|---|---|---|
| 第一步两轮（澄清→PRD） | **同会话** | 第二轮靠第一轮问答上下文写 PRD |
| 第一步→第二步（PRD→设计擂台） | **新会话，只喂 PRD 定稿** | 澄清会话积累了你的取舍倾向信号，带进擂台会让 N 案先天趋同，擂台白开；且长上下文起跑不稳 |
| 第二步内部（design-panel） | 一个会话一条命令 | 方案隔离由插件子代理机制内部保证 |
| 第三步内部（提取→确认→写码） | **同会话** | 确认过的理解要带着写码；拆开重读文档还可能理解漂移 |
| 第三步→第四步（实现→评审） | **必须新会话（硬要求）** | 写码会话满是"我为什么这么写"的辩护性推理，同会话评审=给自己作业打分，烧掉评审 SOP 的全部独立性 |

---

## 第一步：需求澄清 + PRD（人主位）

### 核心认知：第一步不是"让 agent 输出 PRD"

PRD 的核心信息（为什么做、给谁、边界在哪）在你脑子里，不在代码里。让 agent 直接生成 PRD 是**最大陷阱**：它会基于代码现状脑补一个"合理但未必是你要的"需求，后面设计、代码全跟着跑偏，返工最贵。

正确定位：**你把模糊需求倒出来，agent 帮你结构化 + 主动挑漏洞 + 反问澄清。** 跟评审里"建立上下文"同构——先把事实钉死再往下走，跳过这步后面全虚。

### 第一轮 prompt：让 agent 先反问，禁止展开方案

```
我有个新需求，一句话描述：<你的需求>

你的任务不是直接出方案，而是先帮我把需求钉死：
1. 读相关现有代码，理解这块现在是怎么做的（<点名关键文件/模块，或让它自己定位>）；
2. 列出这个需求里【没说清的 / 隐含假设的 / 边界模糊的 / 与现状可能冲突的】
   关键问题，按"不问清就会做错"的优先级排序；
3. 不要在这个阶段给实现方案，只输出澄清问题清单。

我先答这些问题，答完你再起草 PRD。
```

要点：明确禁止它在这一轮给方案（防脑补着往下跑）；让它先读现状代码再提问（否则问的是空泛问题）。

### 第二轮 prompt：据回答出 PRD 草稿（逼它标假设）

```
根据我的回答，起草一份 PRD 草稿，结构：
- 一句话需求
- 背景与动机（解决谁的什么痛、为什么现在做）
- 目标用户与核心场景
- 功能范围（做什么）+ 明确不做（边界）
- 关键验收标准（怎么算完成，可测、可证伪）
- 约束（技术栈 / 兼容性 / 不能动的代码 / 性能与安全要求）
- 风险与待确认项

诚实边界：你不确定的、从代码推断而非我明确说的，一律标【假设】或【待确认】，
不要写成既定事实。这一版是给人改的草稿，不是定稿。
```

**"标假设"是这一步的命门**——agent 不知道的业务道理它只能猜，逼它把猜的标出来，你扫一眼就能纠。这跟评审里的诚实边界一脉相承。

### 回答澄清问题时，你自己也不确定怎么办

核心认知：**"不知道"本身是有效信息，如实标注，绝不为过关硬填。** 瞎编一个答案塞给 agent，它就基于这个错误答案做出"对错误需求的精妙方案"，比没做还糟（与评审禁止把"需运行时验证"写成"已通过"同一条纪律）。

**第一步：先判这个"不确定"能不能当场查证。** 大多数"不确定"其实是"可查证但还没查"，查比问人快、比人脑准。优先序——能查的不问、能试的不猜、剩下真要人拍板的才标待确认：

| 你不确定的是 | 处理 |
|---|---|
| 现状/事实（老接口还有没有消费方、某字段是不是哨兵、线上数据回填没） | **让 agent 当场查**：Grep/git log/读调用链。事实消解不确定，不用问人 |
| 技术可行性（三方接口支不支持批量、某框架行不行） | 标【待验证】，做个最小 spike 试，别靠猜 |
| 业务规则/产品口径（要不要自动恢复、字段对谁可见） | 答案在产品脑子里，查不出来——标【待确认·问产品】 |
| 优先级/范围取舍（边角场景支不支持） | 产品决策——给默认假设 + 标【待确认】 |

**第二步：查不出来的，分阻塞/非阻塞。** 非阻塞（能绕开、推翻了改动也小）：给默认假设让流程往前走，标注"假设可能被推翻 + 推翻代价"。阻塞（不确定就没法设计、做错了返工大）：必须先解决，别带毒进设计擂台。

**回答模板（别只写"不知道"，按这个格式回 agent）：**

```
对 agent 问的"<具体问题>"——

【待确认·问产品】我不确定，倾向【假设】<你的默认推断>，
按这个假设推进设计。
推翻代价：<若假设错了要改什么、改动量>。
```

三要素：**倾向的默认假设 + 标【待确认】+ 推翻后果与代价**。这样 agent 能继续工作，产品确认后你也知道要改哪里。

**一次性问全**：纯 Markdown 无超时续跑，你回一次答就结束本轮。把所有"待问产品"的问题攒成一批一次性丢给业务方，别挤牙膏；自己能查的当场让 agent 查掉，别混进问人的清单。

**边界：不确定太多，说明还没到做 PRD 的时候。** 若 agent 反问清单里一半以上你都答不上，别硬走流程——通常意味着需求还没理解透，需要先做需求调研（看竞品、翻历史代码、找产品聊动机），而不是硬填一堆假设往下做。需求没想清就做，做完大概率错。

## 第二步：方案设计（仅大需求，产品化工具）

PRD 定了之后，大需求直接跑本仓库的设计擂台，**不用手搓**：

- **Claude Code**：`/branch-review-guard:design`
- **Cursor**：从 `/` 菜单选 `/design-panel`（固定视角时传 `--variants N`，详见 `docs/对抗式编排速查卡.md` Cursor 适配节）

机制：并行 N 个 `dsp-designer` 各持价值取向独立成案（互不可见）→ 每案 `dsp-skeptic` 做 file:line 级对抗质证 → `dsp-judge` 裁判打分嫁接。产出：对比表 + 推荐方案 + 嫁接综合方案 + **精炼设计稿**。

**何时用**：动共享代码 / 多条实现路线 / 架构决策 / 高风险需求。性能优化类把已有报告推荐修法钉成方案 A 走单点证伪擂台（速查卡有现成 prompt 生成器）。

**何时不该跳过这步**：中等需求 PRD 够清楚就直接写。设计擂台是给"怎么做有分歧"的需求用的，没分歧硬开是浪费。

**输出落点**：精炼设计稿移入 `docs/`（默认 `docs/<功能>/` 下），第四步评审自动读取。这也是 design-panel `requires: branch-review-guard`、与 `:review` 共享上下文的咬合点。

> 跨平台防漂移契约、固定视角/rubric、子代理上下文注入等细节，查 `docs/对抗式编排速查卡.md`，不在本文重复。

## 第三步：代码实现

按 PRD / 设计文档写码。三条纪律（**已在全局 `~/.claude/CLAUDE.md` 编码价值观节，每会话自动加载，无需写进 prompt**）：

1. **只修根因，不修症状**——别打绕过表象的补丁，先追到底读懂现有代码。
2. **控制改动面**——改一处别引一串连锁；同一逻辑/常量收口到单一定义点。
3. **改逻辑同步改注释/命名**——不留与代码矛盾的过时注释。

**实现 prompt 只装 CLAUDE.md 装不了的任务特定信息**：依据文件、范围钉死、本次修法阶梯定位、验证要求。**统一标准模板**（大需求全占位符照填；中等需求无设计稿时删掉标 ◇ 的行、在依据处改给一段口头方案）：

```
按设计文档实现这个需求。依据（按序完整读）：
- 精炼设计稿：<路径>（执行蓝本）◇
- PRD：<路径>（验收对账；中等需求在此后附一段"怎么做"的口头方案）
- 完整对比报告：<路径>（仅精炼稿有歧义时查证；不得从落选方案取实现
  ——防裁决已放弃的设计混进来）◇

【第一步·先提取后动手】从设计稿/PRD 提取以下四项输出给我确认，
确认前不写一行码：
1. 改动文件清单（模块 + 文件 + 新增/修改）；
2. 落地步骤与依赖顺序（含每步验收点）；
3. 实现硬约束清单（我预填：<已知几条>；有漏的从设计稿补全）；
4. 标【待确认】/【需运行时验证】的点（实现中保留标注，不当已确认写）。

【范围钉死】只实现确认后的清单；设计稿之外的"顺手优化/重构"一律不做
——发现值得改的另列清单给我，不动手。修法阶梯定位：本次是
<L0/L1/L2/additive 纯增量>，不得擅自升级（不许改既有共享方法的
签名/行为）。

【冲突处理】设计稿引用与当前代码不符（行号漂移/结构不存在/假设不成立）
时，停下报告冲突，不自作主张绕过。

【测试】核心逻辑/边界/异常写真断言测试（不写自证桩、不凑假绿）；
新测试类加 @Tag("test")，本地 -P test 全绿；<按项目约定注明测试
是否入库>。

【验证与输出】受影响模块 mvn compile 通过；测试跑不了如实说明，
不写"已验证通过"。输出：改动文件清单 + 与设计稿的偏差说明（无偏差
也写"无"）+ 运行时验证项清单（从设计稿原样带出）。不执行 git commit。
```

模板要点：

- **先提取后动手是标准机制，不是可选变体**——范围/步骤/约束不要人工从设计稿提取（精炼稿本来就是写给 agent 读的），agent 提取、你扫清单确认，既省力又能当场发现它漏读；硬约束"我预填几条 + 有漏的补全"是双保险。
- **范围钉死 + 阶梯不升级**是防 agent 顺手扩大改动面的两道闸（对应"最好的代码是不用写的代码"与"尽量不改共享代码"）。
- **冲突停下报告**防设计稿过期时 agent 硬绕；**@Tag** 是评审实测抓过的假绿坑。
- **三份文档给阅读优先级**，完整对比报告降为查证用且明禁从落选方案取实现；"运行时验证项原样带出"让设计→实现→评审的诚实边界连续传递，`:review` 对账直接可用。

> 同步提醒：本模板与评审 SOP 第 5 步"修复 prompt"共享骨架块（范围钉死/取证纪律/如实输出/不 commit），属**有意冗余**（各自复制即用）；改任一处的共享块措辞，检查另一份是否需同步。

提交纪律走项目 memory：`git commit -- <paths>` pathspec 分批提交，**禁止 reset/add 重排暂存区**（会冲掉 IDE 变更列表分组）；可用个人 `/commit` skill 自动学风格 + 拆批。**只 commit 绝不 push** 除非明确要求。

## 第四步：评审 → 接评审 SOP

代码写完，按 `docs/2+2+1合裁评审-SOP.md` 的准入表选档：

- 日常小改 → 单模型 `:diff`
- 提测前 / 重要重构 → 1+1（评审 + 单证伪）
- 上线关键分支 / 涉资金·权限·数据一致性 → 全套 2+2+1 + 人工兜底

评审发现的误报当轮沉淀成 calibration，让下一轮更便宜（飞轮）。本步不展开，见评审 SOP。

---

## 诚实边界

- **需求层 agent 是辅助不是产出方**。PRD 的源头是你，agent 只结构化和质询。让它直接生成 PRD 必脑补，是全链路最大返工源。
- **别过度流程化**。准入表是硬约束：小改直接写，中等只补验收标准，只有大需求才值得在第二步开设计擂台。流程是给"有分歧/高风险"的需求省返工的，没分歧硬上是负收益。
- **PRD 的【假设】/【待确认】必须人确认后再进第二步**。带毒的需求进设计擂台，只会得到"对错误需求的精妙方案"。（澄清关卡"一次性问全"纪律见第一步，不重复。）
