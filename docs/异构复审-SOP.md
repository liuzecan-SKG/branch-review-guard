# 异构复审 SOP — 用"不同模型 + 人"补单模型盲区

> 面向用 branch-review-guard 做分支评审的开发/评审同学。这是**使用实践**，不改插件、不写进自动流程——照着做即可。配套机制见 `skills/branch-review-guard/SKILL.md ## 使用建议：异构复审（强独立召回）`。

## 一句话

插件一次评审内部是**同一个模型自演自验**（子代理固定 `model: opus`）；想要真正的"独立第二意见"，靠**主会话换到非 Claude 家族的模型跑多轮 + 人工复核**——这才是同事 114-agent + 双人 2/2 里"强独立"的部分。

本文含两个方向：**① 异构复审（第 1-5 步，治召回/降漏报）**——换推理者把漏掉的捞回来；**② 精度复核（`## 另一个方向：精度复核`，治精度/降误报）**——换纯证伪立场把误报揪出来。两者都靠"独立立场"，一个补漏、一个纠错，可分别或配合用。

> ⚠️ 澄清一个易混点：Claude Code 支持给子代理 per-agent 指定模型，但取值只到 **Claude 家族**（opus/sonnet/haiku/fable）。本套件已把承重与各维度子代理统一升到 opus（升绝对质量）。但**家族内不同档位是弱异构**——opus/sonnet/haiku 同源、错误相关性高，混用换不来正交盲区。真异构（错误不相关）**必须跳出 Claude 家族**，只能在主会话层换模型（若你的部署能切 GPT/Gemini 等）或靠人。

## 为什么需要（先想明白再做）

评审漏报分两类失败，对策不同：

| 失败类型 | 例子 | 同模型能补吗 | 靠什么补 |
|---|---|---|---|
| **注意力/推理路径没聚焦** | 大文件中段的 N+1、孪生接口防护不对称 | 部分（loop-until-dry、交叉确认有效） | 换模型收益明显（注意力分布不同） |
| **根本不知道这条业务道理** | 墓碑哨兵是防穿透、单设备单激活是产品要求 | ❌ 换多少模型都没用 | 人喂领域知识（不变式台账 / 需求文档） |

**插件内的 loop-until-dry 和 P0/P1 交叉确认，都只能补第一类的一部分**——因为它们是同底模，同模型盯不到的地方跑多少轮都盯不到。异构复审补的是第一类的剩余部分（换模型）+ 第二类（换人）。

## SOP 四步

### 第 1 步：主审

用当前模型正常跑一次，得**报告 A**：

```
/branch-review-guard:review
```

### 第 2 步：换模型

```
/model
```

选一个**架构或厂家不同**的模型——同家族（如同一厂商的大小版本）盲点太接近，换了约等于没换。具体可切哪些以 `/model` 实际列出为准；挑差异最大的那个。

> ⚠️ 这一步是异构的关键。换成"更强的同款模型"提升的是能力、不是独立性；换成"不同架构的模型"才带来正交的盲区。
>
> ⚠️ **一个要知道的细节**：本套件子代理 frontmatter 固定 `model: opus`，**优先级高于主会话模型**（Claude Code 四级优先级：环境变量 > 单次派发参数 > frontmatter > 主会话）。所以你在主会话 `/model` 换个模型，**维度子代理仍跑 opus**、不会跟着变——真正换到的是"编排器本身"的推理。若要让子代理也换（比如整体切非 Claude 家族做强异构），需用 `CLAUDE_CODE_SUBAGENT_MODEL` 环境变量覆盖（它优先级最高），或临时改 frontmatter。日常异构复审靠"编排层换模型 + 人工复核"即可，不必每次改环境变量。

### 第 3 步：换模型后补审

- **省成本版**（推荐日常用）：只补最吃推理多样性的三个维度——

  ```
  /branch-review-guard:review --dimensions semantic,security,bug
  ```

  业务语义、安全、正确性这三类最值得换个模型看第二眼；DDL/i18n/格式这些确定性强的维度换模型收益低，不必重跑。

- **彻底版**（关键分支、上线前）：整份重跑，得**报告 B**。

> ⚠️ **防"同 SHA 无需重跑"投机**：换模型后 commit 没变、diff 字节和上次一样，agent 可能自作主张说"同一 SHA、无信息增益，不重跑，改为复核上次结论"——**这是错的，别接受**。异构复审的全部价值就在"同一份代码、换个模型再看一遍"，SHA 不变正是常态。看到这种说法，明确回它：**按换后的模型真实重跑维度评审 + 对抗验证，不许复用上一个模型的结论**。（插件已在 `SKILL.md ## 每次评审都是独立执行` 写死这条纪律，但运行时若仍出现投机，手动纠正。）

### 第 4 步：对照合并

把两份报告并排比：

| 情况 | 含义 | 处理 |
|---|---|---|
| **A、B 都报同一问题** | 两个独立推理者都命中 | **高置信**，优先修 |
| **只有 A 报 / 只有 B 报** | 异构捞回的候选 | 标"单源、待人工复核"——正是换模型的价值所在，别丢 |
| **A、B 结论冲突**（一个说是 bug 一个说正例） | 触及模型判断分歧 | 人来裁，通常是业务语义的灰区 |

### 第 5 步：人工兜底（最强异构）

对 Top 的 P0/P1，自己或拉一个同事**独立看一遍**（别先看报告结论，从代码看起）。

**人是当前能用的最强"不同模型"**——这就是同事"双人 2/2"的精髓。尤其是第二类失败（业务道理），只有懂这块业务的人能兜住。

## 什么时候值得做

| 场景 | 建议 |
|---|---|
| 日常迭代 diff | 插件单模型 + `--thorough` 够用，不必异构 |
| 提测前/上线前关键分支 | 做第 1-4 步（省成本版补审足够） |
| 涉及资金/权限/数据一致性的核心链路 | 全套五步，第 5 步人工兜底不能省 |
| 已知模型对某类业务反复漏（如某种不变式） | 把该类不变式写进领域台账（见下），比反复换模型更根治 |

## 另一个方向：精度复核（独立证伪，降误报）

前面五步治**召回**（换个推理者，把漏掉的问题捞回来）。反过来还有一个同样有效、成本更低的动作治**精度**：**拿一份已产出的评审报告，换个纯证伪立场重新审一遍，把误报揪出来**。

**为什么有效**：原报告是在"找问题"的立场下产出的，带确认偏误——一旦找到一堆问题，对抗验证也容易顺着"这些都对"往下走。换一个**纯证伪立场**（默认每条都是误报、积极找反证）独立重跑，很多"看着成立实则脑补"的发现会现形。**这不需要换模型**——独立会话 + 立场翻转本身就是一种弱异构（"立场独立"），实测能揪出同模型一次编排没降住的误报（如"机制错配当绕过""正交语义当矛盾""安全闸门当降级缺陷"这几类）。

**怎么做**：把评审报告里的发现清单（每条带 file:line）抽出来，喂给一个独立会话，用下面的证伪 prompt 骨架跑。它是"单点证伪"剧本（对抗式编排速查卡）的标准应用：

```
【任务】对下面评审报告标出的问题，逐条对照真实代码做单点证伪，
判定"真实存在(CONFIRMED)/削弱(WEAKENED)/误报(REFUTED)"。一气呵成不回问。

【基线与取证纪律（硬约束）】
- 基线：git diff <base>...HEAD（三点）。先 git fetch，Read 读全文、以当前代码为准，
  不信已过期的行号假设。
- DLP：部分文件 Read 也可能返回 %TSD-Header 密文，Bash grep/rg 一律可能假零。
  读到密文改用 git show/git diff。否定型结论必须标 [Read]/[git]，禁用 grep"没搜到"下结论。
- 裁决门槛（先公布防揣摩）：怀疑者默认立场=积极找反证证明"这是误报"；
  但推翻/削弱一条必须给 file:line 级机制反证，"感觉不严重/可能有兜底"不算、按维持计。
  · CONFIRMED：尽力证伪未果、触发路径真实可达。
  · WEAKENED：存在但严重度高估或有部分兜底（给 file:line）。
  · REFUTED：有 file:line 反证证明前提不成立（其实已有 try/catch/校验/索引/该字段本不归 X 管…）。
  · 拿不准一律标"需运行时验证"，不伪造成 CONFIRMED/REFUTED。
- 只判"问题在分支代码里是否真实"，不做修法设计。

【共享事实包】<把报告的发现逐条列出，每条：编号 + 命题 + 已定位的 file:line 证据>

【派发】P1/承重项每条派 2 视角（①证据核实 ②触发路径），P2 每条派 1 合并怀疑者；
每个怀疑者任务书注入：本条命题 + file:line 事实 + 取证纪律 + 裁决门槛。收尾派 1 个完整性批评家，
核对：有无"感觉不严重"式无证据推翻、有无 grep 假零未 Read 复核、有无该标"需运行时验证"却写死 CONFIRMED/REFUTED、是否逐条给了结论。

【输出】逐条一行表（编号|发现|结论|关键 file:line 证据或反证|严重度是否调整）+ 附录（每条 3-4 句说明证伪尝试）。
```

**证伪结果的两个去处**：
1. **误报模式沉淀成 calibration 规则**：反复出现的误报类型（如上面三类）走 `distill`/`rule` 闭环，落 `rules/` 或项目本地 `branch-review-rules/`，让插件**下次少犯同样的误报**——这比改 prompt 更根治。本套件已把这三类通用误报固化为 `baseline/` 的 `calibration-mechanism-mismatch-not-bypass` / `calibration-orthogonal-not-contradiction` / `calibration-security-gate-not-degradable`（默认开）。
2. **修正报告定级**：REFUTED 的从阻塞清单剔除、WEAKENED 的降级，得到一份精度更高的最终清单。

> ⚠️ 边界：精度复核只查"报告已标的对不对"（降误报），**不查"报告没标的有没有漏"**（那是召回，靠前面五步）。别把"清单验真"当成"评审无漏"——两轮全 CONFIRMED 也可能有真问题两轮都没想到去质疑。且它仍是弱独立（同模型可能有共同盲区，有些误报两轮都识不破），关键分支该配人工复核。

## 配套：领域不变式台账（治第二类失败的根）

换模型治不了"不知道业务道理"。真正的解法是把团队已知的业务不变式**沉淀成清单**喂给评审：

- 位置：被评审项目根的 `branch-review-rules/`（插件会 best-effort 全量叠加，独立于规则包开关）。
- 内容：单设备单激活、登出墓碑哨兵语义、状态机单向、某字段的 null 是哨兵不是脏数据……每条写清"不变式是什么 + 哪些路径必须守 + 违反的后果"。
- 效果：业务语义维度（`bru-business-invariant`）评审时读到这些，就能从"猜不到"变成"按台账核对每条路径"。

这是"人预置领域知识"的可操作形态——**人不用从头看代码，人把'道理'交出来，模型负责核对每条路径**。一次沉淀，长期复用，比每次评审都换模型划算。

## 诚实边界

- 异构复审提升的是**召回**（少漏），代价是**多跑一遍 + 人力**。它不保证零漏——第二类失败里团队自己都没意识到的不变式，谁也兜不住（未知的未知）。
- `✓✓交叉确认`（插件内同底模 opus）≠ 异构复审的"A、B 都报"——前者弱独立（同源模型错误相关），后者强独立（换非 Claude 家族）。报告里别把两者混为"已独立验证"。
- 换模型不是万能药：确定性维度（DDL 缺失、i18n key、格式）换模型没意义，那些该靠规则/工具（未来的 Semgrep 层），不靠多模型。
