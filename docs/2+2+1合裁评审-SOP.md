# 2+2+1 合裁评审 SOP — 双模型评审 × 双证伪复核 × 合并裁决

> 面向用 branch-review-guard 的开发者。这是**使用实践**，不改插件。它是 `docs/异构复审-SOP.md` 两个方向（异构复审治召回、精度复核治误报）的**组合完整版**：2 份评审 + 2 份证伪 + 1 次合裁。源自 2026-07-13 对 `a7c5d7e0c` 未提交 diff 的实测（两层均有实打实收益：跨模型捞回 5+1 条漏报，双证伪纠出 ~10 处误报/高估，零误杀）。

## 一句话

**双模型平行评审补漏（召回），每份各配一次立场翻转证伪纠错（精度），第三会话合并裁决分桶，人只在处置关卡出现一次，每轮误报必须沉淀成规则让下一轮更便宜。**

## 准入（先判值不值，别默认全套）

| 场景 | 用法 |
|---|---|
| 日常迭代 diff | 单模型 `--thorough` 即可，**不进本 SOP** |
| 提测前 / 重要重构收尾 | 1+1：单评审 + 单证伪，跳过双模型 |
| 上线关键分支 / 涉资金·权限·数据一致性 | **全套 2+2+1**，且第 5 步人工兜底不能省 |

全套成本 ≈ 5 个会话（2 评审 + 2 证伪 + 1 合裁）。摊薄成本的唯一方式是第 5 步沉淀——本次实测若三条 calibration 提前存在，两份报告的 Top 阻塞项各少 5~6 条。

## 会话拓扑与执行细节（三档怎么开会话）

**地基：会话隔离 = 立场独立。** 证伪的价值来自"没看过评审找问题时的推理链"——同会话让它复核自己的报告等于给自己作业打分，只会确认不会翻案。实测的全部翻案（F7 降级、N3 近乎推翻等）都发生在全新会话 + 立场翻转下。会话数 = 需要几个互相看不见的独立立场。

### 档位一：日常 diff（1 个会话）

项目会话直接 `/branch-review-guard:diff`，摸到共享代码/事务/权限加 `--thorough`。插件内置对抗层（bru-skeptic 三视角 + 反向证伪 + critic）已是"半次证伪"，报告照着修。义务两条：`⚠单源` 的 P0/P1 自己扫一眼 file:line 再动手；遇到误报当场 `:rule` 沉淀。

### 档位二：提测 1+1（2 个会话）

- 会话 1（评审）：项目里跑 `:diff`/`:review`。
- 会话 2（证伪）：**新开会话**（同项目目录无妨，关键是上下文全新），喂下面的证伪 prompt 模板。
- **证伪不换模型**——1+1 买的是立场独立（弱异构），换模型是档三的事。
- 不需要第三会话合裁：证伪报告自带修正后清单，人直接处置。
- **升档信号**：证伪翻掉评审 P0/P1 的一半以上 → 该 diff 语义复杂度超出单模型把握，升 2+2+1。

**证伪 prompt 模板（档二会话 2 / 档三会话 3、4 通用，`<>` 处按次填）：**

```
【任务】对下面评审报告标出的发现，逐条对照真实代码做单点证伪，判定：
CONFIRMED（真实存在）/ WEAKENED（存在但定级高估或有部分兜底）/
REFUTED（误报）。一气呵成不回问。你没有参与该评审，不要采信报告的
推理过程，只认当前代码。

【被复核报告】<报告路径>（先完整读它）
【基线】与报告同口径同 SHA：<diff 模式 = git diff + git diff --cached /
branch 模式 = git diff <base>...HEAD 三点>。以 Read 读到的当前代码为准，
不信报告里可能过期的行号。

【分层派发（必须，省一半成本）】
- 命令可核验类（git 状态/符号存在性/文件计数/零引用）：编排器直接核验，
  不派怀疑者。
- 语义类 P0/P1（越权/不变式/降级/事务/口径）：每条派 3 镜头怀疑者
  （①证据核实 ②触发路径 ③规则校准），默认立场 = 积极找反证证明它是误报。
- P2/Nit：每条 1 个合并怀疑者或编排器自核。
- 报告声明的"正例/零发现"抽 2-3 条派「假设证伪」镜头（防漏报侧确认偏误）。

【裁决门槛（先公布防揣摩）】
- 推翻/削弱必须给 file:line 级机制反证；"感觉不严重/可能有兜底"不算，按维持计。
- 拿不准一律标"需运行时验证"，禁止伪造成 CONFIRMED/REFUTED。
- 只判"问题是否真实存在"，不做修法设计。

【取证纪律】<按环境填，如：本仓库 Bash grep 对 .java 有假零，否定型结论
只认 Grep 工具 / git show / Read，并标注 [Read]/[git]>

【输出】落盘 branch-review-reports/verify-<模式>-<SHA>-<日期>-<模型名>.md：
逐条裁决表（编号|发现|原定级|裁决|关键反证 file:line|定级处置）+
修正后清单（该修/降级/翻案/待确认四组）+ 诚实边界（与原报告同底模的
弱独立声明、剩余未验证项）。
```

### 档位三：上线 2+2+1（5 个会话 + 人）

```
会话1: 模型A 评审 ─┐            ┌─ 会话3: 证伪A ─┐
                    ├─(可并行)──┤                 ├─→ 会话5: 合裁 ─→ 人: 处置
会话2: 模型B 评审 ─┘            └─ 会话4: 证伪B ─┘
```

- 1/2 可两窗口同时跑，3/4 同理；串行依赖只有"评审→它的证伪→合裁"。
- 会话 2 换模型两个坑：①子代理 frontmatter `model: opus` 优先级高于主会话，`/model` 只换编排层（日常够用；全换用 `CLAUDE_CODE_SUBAGENT_MODEL`）；②同 SHA 必须真实重跑，"diff 没变无需重跑"是投机，顶回去。
- 会话 3/4 直接复用档二的证伪 prompt 模板。
- 会话 5 也开在项目目录——冲突裁决常要亲眼 `git diff` 一锤定音；第一动作永远是核四份文件同 SHA 同模式。
- **人工兜底的正确姿势**：对 A 桶资金/权限类 Top P0/P1，**先读代码、后看报告结论**（顺序反了会被锚定），每条 10~15 分钟。这是唯一兜得住"第二类失败"（模型不知道的业务道理）的网。

**会话 2 评审启动 prompt（`/model` 换到跨厂家模型后新开会话）：**

```
对 HEAD <SHA> 执行 /branch-review-guard:<diff|review> --thorough，
与模型A那轮完全同口径。本次是异构复审：同一 SHA 换模型重看正是目的——
不许以"同 SHA / diff 未变 / 已有报告"为由跳过或复用既有结论，必须
真实重跑全部维度评审与对抗验证。
报告文件名带模型名：branch-review-guard-<模式>-<SHA>-<日期>-<模型名>.md
```

**会话 5 合裁 prompt：**

```
【任务】对同一份 diff 的四份报告做合并裁决，产出单份分桶清单。一气呵成。

【输入】
- 模型A 评审：<路径>　/　模型A 证伪：<路径>
- 模型B 评审：<路径>　/　模型B 证伪：<路径>

【第一步·口径核验（不过不做）】四份必须同 SHA、同模式（diff/branch）、
同 --thorough 档位；不一致立即停下报告差异，禁止硬合不可比对象。

【第二步·各自修正定级】按各自证伪结果：REFUTED 剔除、WEAKENED 降级、
"需运行时验证"原样保留、复核未覆盖的条目标注"未经第二立场"。

【第三步·跨模型配对】按 file:line + 问题机制配对，不按标题字面
（两模型对同一问题的表述会不同）：
- 双报且双 CONFIRMED → 高置信；
- 单源且该侧 CONFIRMED → 异构捞回（标注来源，别丢）；
- 双方冲突 → 列为待人裁，必要时亲自读代码给裁决建议。

【第四步·分五桶输出】
A 修（双确认硬事实 + 证据硬的单源捞回）/ B 待产品业务确认 /
C 未经第二立场的存量高危（附补验建议）/ D 翻案降级（只修定级默认不修，
残余廉价改进单列供选）/ E 沉淀候选（误报模式→calibration、
异构捞回的漏报模式→finding、业务道理→不变式台账）。

【输出】落盘 branch-review-reports/merged-adjudication-<SHA>-<日期>.md，
末尾附诚实边界：各层独立性强弱（同家族=弱异构要写明）、证伪只验精度
不验漏报、运行时项不得升格为"已通过"。
```

### 三档对照

| | 会话数 | 异构来源 | 裁决者 | 单轮成本 |
|---|---|---|---|---|
| 日常 diff | 1 | 插件内置对抗层 | 报告自身 | 顺手 |
| 提测 1+1 | 2 | 立场翻转（弱） | 证伪报告 | ~半小时 |
| 上线 2+2+1 | 5 + 人 | 跨模型（强）+ 立场 + 人 | 合裁会话 + 人 | ~半天 |

**升降档只看一件事：这条 diff 错了会怎样。** 自己能兜住→档一；返工可控→档二；上线背锅→档三。三档共享同一义务：误报当轮沉淀，成本逐轮变便宜。

## 流程五步

### 第 0 步：前置（10 分钟，决定后面一半的信噪比）

1. **先查项目本地 `branch-review-rules/` 是否已装载历史豁免**——已知的工作流特例（本地 pom 隔离、测试不入库、个人文件）不先沉淀，每个模型每轮都会当 Top 阻塞项报一遍。
2. **钉死口径并写进文件名**：`diff`（工作区）还是 `branch`（三点）？两份评审必须同口径、同 SHA、同 `--thorough` 档位，否则第 3 步不可比（实测踩过：branch 报告混进 diff 对照，交叉确认全部作废）。
3. 命名与登记纪律：`<模式>-<SHA>-<日期时间>-<模型名>.md`，四份文件在一个清单里登记配对关系。**模型名必须进文件名**——事后没人记得哪份是谁跑的。

### 第 1 步：双评审（召回层，可并行）

两个**尽量异构**的模型各跑一次同口径 `/branch-review-guard:diff --thorough`（或 `:review`）。

- 异构优先级：跨厂家 > 跨家族 > 同家族不同档（弱异构，收益打折）。
- 防投机：同 SHA 换模型必须真实重跑（`SKILL.md ## 每次评审都是独立执行`）。
- 两份评审互不可见对方结论。

### 第 2 步：双证伪（精度层，可并行，**分层派发省一半钱**）

每份报告各起一个独立会话，跑 `docs/异构复审-SOP.md ## 精度复核` 的证伪 prompt。关键效率规则——**按发现类型分层**：

| 发现类型 | 派法 | 依据 |
|---|---|---|
| 命令可核验类（git 状态、符号存在性、文件计数） | 编排器直核，**不派怀疑者** | 实测这类全部坐实、零翻案，3 镜头是纯浪费 |
| 语义类 P0/P1（越权、不变式、降级、口径） | 3 镜头怀疑者（证据核实/触发路径/规则校准） | 实测翻案/降级全部发生在这层 |
| P2/Nit | 1 个合并怀疑者或直接跳过 | 定级本来就低，证伪边际收益小 |

诚实边界：证伪与评审若同底模，是**立场独立（弱异构）**——能纠过度解读/越界/触发路径，纠不了共有语义盲点；报告里必须写明。

### 第 3 步：合并裁决（第三会话，给四份文件路径即可）

顺序固定：

1. **先核口径**：四份同 SHA 同模式才继续（防第 0 步的配对错误漏网）。
2. **各自修正定级**：REFUTED 剔除、WEAKENED 降级、"需运行时验证"保持原样。
3. **跨模型对照**（按"file:line + 机制"配对，不按标题字面）：
   - 双报且双 CONFIRMED → **高置信，直接进修复清单**
   - 单源且该侧 CONFIRMED → **异构捞回**，标注来源、别丢（这是换模型的全部价值）
   - 双方冲突 → 留给人裁，通常是口径或业务灰区
4. **分桶输出**（固定五桶，处置关卡按桶下指令）：
   - **A 修**：双确认硬事实 + 单源捞回中证据硬的
   - **B 确认**：待产品/业务/BI 拍板项
   - **C 补验**：未经过第二立场的存量高危（如另一口径报告的发现）
   - **D 翻案**：定性被复核推翻/削弱/越界/降级——只修正定级默认不修，残余廉价改进单列供选
   - **E 沉淀**：误报模式→calibration、异构捞回的漏报模式→finding、领域道理→不变式台账

### 第 4 步：处置关卡（人，唯一必须出现的点）

逐桶给一句话指令（修 / 豁免入规则 / 待确认 / 忽略 / D 桶哪几条顺手修）。人不看代码细节，只做取舍——代码事实已经过两层核验。

### 第 5 步：执行 + 沉淀（回项目会话，一个 prompt 收尾）

修复 prompt 三条硬纪律：**范围钉死**（清单外一行不碰，工作区可能有进行中改动）、**破坏性 git 操作先留底**（restore 前存 diff 备份）、**不自动 commit**（整理完暂存区停下给人看）。

**修复 prompt 模板（处置关卡定完桶后，回项目会话）：**

```
我要按合并裁决报告执行收尾。依据（先完整读）：<合裁/证伪报告路径>

【范围钉死】只做我点名的条目：<列出桶/编号，如"A 桶全部 + D1/D6"）>；
报告里其余条目一律不动。工作区可能有进行中的改动，清单外文件一行不碰。
全程不执行 git commit——整理完暂存区停下来给我看。

【破坏性操作留底（硬约束）】
- git restore / git rm --cached 等操作前，先把当前 diff 原文存到
  <备份路径，如 branch-review-reports/local-backup-<日期>.diff>；
- unstage 一律用 git restore --staged，只动暂存区、不删工作区文件。

【逐项清单（按序，每项带定位）】
1. <动作 + 文件路径 + 约束，如"只动超期分支，不改既有豁免逻辑">
2. ...

【取证纪律】<环境假零说明，同证伪 prompt>

【输出】逐项完成状态表 + git status --short 全文 + 改动文件清单；
做不了的如实标注原因，不写"已验证通过"。
```

**沉淀 prompt（同会话顺手带上，或单独跑）：**

```
另外执行两类沉淀：
1. 项目特例豁免 → 在 branch-review-rules/ 新建 calibration 文件
   （frontmatter 参照已装插件 rules/ 的 schema）：<逐条列豁免内容与理由>
2. 通用误报/漏报模式 → 走 /branch-review-guard:rule 生成草稿
   （finding=以后主动查 / calibration=以后豁免降噪）：<逐条列模式+证据样例>
草稿只落本地，我人工确认后先移入 branch-review-rules/ 试用（记 LEDGER），
服役出战绩再晋升插件仓库。
```

沉淀三路：项目特例 → 目标项目 `branch-review-rules/` calibration（扁平放根、`pack: local`）；通用误报/漏报模式 → `:rule` 草稿 → 人工确认 → 先落 `branch-review-rules/` 试用 → 战绩达标（命中 ≥3、存活率 ≥2/3）晋升插件仓库 `rules/discover-new/`；业务道理 → 领域不变式台账。**跳过这步，全套成本永远摊不薄。**

## 诚实边界

- 全流程若都在 Claude 家族内，语义盲区共享——两份评审 + 两份证伪都没报 ≠ 没有问题。第二类失败（不知道的业务道理）只有人和不变式台账兜得住。
- 证伪只验"已报的对不对"，不验"没报的漏没漏"；双评审重合度高也不是无漏证明。
- 运行时项全程只标"需运行时验证"，合裁不得升格为"已通过"。
