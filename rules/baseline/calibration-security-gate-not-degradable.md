## 识别要点

- 发现形如"X 调用无 try/catch/fallback，抖动时整页/整请求 500，应降级"。
- 在批量指控"核心链路缺降级"时，尤其容易把**安全校验闸门**也一并算进"该降级却没降"。

## 取证方式

- 命中此类发现时，**先 Read 判定 X 的性质**——它产出的是"装饰/展示性数据"（点赞数、趋势图、演示位）还是"安全/授权/归属校验闸门"（鉴权、越权归属校验、名额/资格校验、数据可见性判定）。
- 判据：X 失败时若"继续往下走"会导致**越权、看到不该看的数据、绕过资格限制**，则 X 是**闸门**——它失败时**就该拦停整请求**，这是 fail-close 的安全设计，不是缺陷。

## 校准动作

- X 是安全/授权闸门 → **"该调用缺降级"这条发现对 X 不成立，剥离/越过**（闸门 fail-close 是预期行为，降级反而是漏洞）。
- **但装饰性数据的降级缺失仍然维持**：同一批里真正的展示性字段（趋势图、点赞数、演示补救）失败拖垮整页，那是真缺陷、按原级别保留。
- 本校准只把"安全闸门"从"缺降级"的指控里摘出来，不放松对装饰性数据降级缺失的判定。区分标准 = 失败继续走会不会越权，不是"是不是 RPC"。

## 由来

实测：一次评审把 `getAllFriendListByUserId`（亲友归属校验闸门，失败=不知谁能看→必须拦）和 `loadWxappTrendOverview`（趋势图装饰位，失败可降级）一起算进"核心 RPC 缺 try/catch 降级"。证伪拆分：前者是安全闸门、整页失败是预期（非缺陷），后者才是真正"可降级却没降"的 P1。
