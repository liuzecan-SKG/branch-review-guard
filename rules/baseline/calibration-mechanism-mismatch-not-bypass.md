## 识别要点

- 发现形如"字段/接口 A **绕过**了 X（过滤器/权限 map/校验器/脱敏配置/治理清单）"、"A 未经 X 处理，与 B 口径分叉"。
- 典型场景：某响应字段没走某个 shareConfig / fieldFilter / 脱敏 map / 白名单，而**同响应里另一些字段走了**——于是被判"绕过 X、口径不一致"。

## 取证方式

- 命中此类发现时，**先 Read X 的实际管辖清单**（X 的枚举定义 / 配置项 / 过滤字段列表），确认 A 是否本就在 X 的射程内。
- 若 X 的定义里**根本没有 A 这类对象**（X 历来只治理另一组对象，A 从不在其列），则"绕过"命题的前提不成立——A 没走 X 不是"绕过"，是**A 本就不归 X 管**（机制错配）。取证标 `[Read] X 定义清单无 A`。

## 校准动作

- 前提被证伪（X 不治理 A）→ **原"绕过 X / 口径分叉"发现直接越过**，不计入发现，或改写为"X 不治理 A，两者无治理关系"。
- **但要剥离独立成立的子问题**：若 A 本身确有问题（如明文手机号暴露、缺独立授权），那是 A 自己的缺陷，**按 A 的真实问题另开一条**（不挂在"绕过 X"这个错框架下）——不要因为主框架被证伪就把 A 的真问题一起丢掉。
- 本校准只降"错误归因（脑补出不存在的治理关系再指控绕过）"的噪音，不放松对 A 真实缺陷的判定。

## 由来

实测：一次评审报"健康页装饰字段(佩戴态/告警数/明文手机号)绕过 shareConfig 逐字段过滤"，证伪发现 shareConfig 历来只治理 10 类健康指标卡片、那三个字段本不在其射程——"绕过"是脑补的治理关系。剥离后仅"明文手机号暴露"是独立成立的隐私问题。业务语义维度擅长主动联想找漏报，副作用是有时脑补不存在的约束再指控被违反，本规则专降这类。
