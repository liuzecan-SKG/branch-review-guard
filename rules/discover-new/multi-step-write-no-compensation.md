## 识别要点

- 一个业务动作拆成 ≥2 步写（本地 DB + 跨服务 RPC / MQ / Redis 占位），追问三件事：后步失败时前步留下什么？有没有补偿/对账？重试能不能自愈？
- 高危形态：①先消费一次性资源（验证码 delete、SET NX 占位、令牌）再做可失败的关键调用；②先落库（含发 MQ）再调跨服务创建，失败留孤儿行；③失败重试路径上有幂等判断提前 return，把"半完成态"当"已完成"跳过修复。
- 多表删除/清理无事务包裹也属本条（任一步抛错半删）。

## 取证方式

- 给出每步写操作的 file:line 与顺序，指出失败注入点及其后果（孤儿行/烧掉的窗口/作废的码）；核对是否存在补偿代码或定时对账任务（用 Read 确认，不凭猜测说"应该有"）。
- 幂等短路要给出提前 return 的分支行号，并说明它拦截的恰是自愈重试。

## 修法

- 顺序原则：可回收/可重试的步骤在前，不可回收资源消费与"点火"步骤放最后；一次性资源用"预扣+确认/超时返还"两段式。
- 跨服务多步写补对账出口（定时任务或重试队列）；幂等判断区分"已完成"与"半完成"，半完成态放行修复路径。
- 多表删除包进事务，MQ 后置到提交后。
