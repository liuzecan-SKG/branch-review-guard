# Design Panel

本 skill 是一个**设计编排器（orchestrator）**——评审侧的 `branch-review-guard` 守「出口」（已写代码能否发布），设计侧的本 skill 守「入口」（要写的代码该怎么落）。它把一次「需求方案设计」拆成多个**互不知情的设计代理**从不同价值取向各出一版，再用独立**怀疑者**对每案的承重论断做 file:line 级对抗质证，最后由**裁判**打分、嫁接，产出一份中文设计文档。

设计原则（综合「多智能体编排」对抗误判与漏判的实证）：

- **方案先于代码**：方向错的成本是改代码的十倍。设计阶段花 5-20 倍 token 换「多视角独立成案 + 对抗质证」，比单 agent 一把梭更可能逼近项目最优解。
- **隔离产出多样性**：方案代理**互相不可见**——看见对方结论会诱发趋同（anchoring）；隔离使成本随 N 线性可控。隔离是产品，不是优化。
- **承重论断可证伪**：每个方案必须显式给出 ≤4 条「本方案成立所依赖的、可被反驳的事实性判断」，每条带 file:line 证据。怀疑者只攻这些承重墙。
- **单挑战者 + 四件套护栏**（非评审侧的 3 票聚合）：设计裁决最终由人拍板、误杀不阻塞发布，故对抗强度低于评审侧；但单挑战者的「假维持」会腐蚀横向排名，故配四道护栏对冲。
- **诚实边界**：区分事实/推测/待运行时验证；性能、并发、迁移类论断只允许写「需运行时验证项」，**禁止写「已验证通过 / 性能无影响」**。本 skill 的诚实边界由编排器在定稿前用固定 checklist 独立核查。

## 可移植性（自包含，不依赖任何特定 Agent）

本 skill 是纯「Agent 规则 + Git/读文件命令」，可在 Cursor / Claude Code / Codex CLI / 其他能读文件 + 执行 Git 的 Agent 中运行。

- **支持子代理的 Agent**（如 Claude Code 插件形态、Cursor 的 Task）：P2 并行派 N 个 `dsp-designer`、P3 每方案 1 个 `dsp-skeptic`、P4 一个 `dsp-judge`，上下文隔离、互不挤占。**子代理并行是本 skill 价值成立的机制前提**。
- **不支持子代理的 Agent**：顺序多轮，同一套 prompt；但**设计侧的隔离即产品**，顺序模式下方案独立性会弱化（见 `## 降级策略`，与评审侧「结果一致只是更慢」口径不同，不得混用）。

本套件的发布载体是 Claude Code 插件（`.claude-plugin/`）；非 Claude Code 的 Agent 经安装器（`install/SKILL.md`）落地到 `tools/<name>/` 或直接读 `skills/<name>/SKILL.md`。本 skill 依赖 `branch-review-guard`（共享 `rules/` 与 `reports/` 目录），`manifest.json` 中声明 `requires: ["branch-review-guard"]`。

## 规则机制（只消费现有规则包，不改 schema）

本 skill **栈无关核心**：设计视角的价值来自价值取向差异，不绑定框架。技术栈/项目特有的「坑」与「降噪校准」复用 `branch-review-guard` 的可插拔 **`rules/` 规则包**——把已启用栈包中 `type: finding` 的规则作为「设计约束」注入 designer 与 judge。

- **规则包位置**：随 `branch-review-guard` 安装（canonical：`tools/branch-review-guard/rules/`）；启用哪些包由 `rules/config.yaml` 控制。schema 与消费方式见 `rules/README.md`。
- **项目本地规则叠加**（v0.5.0+，与评审侧同源）：除插件自带 `rules/` 外，designer/judge 叠加读取**被设计项目根**的 `branch-review-rules/`（项目主人自放、纯本地不入库，建议写 `.git/info/exclude`）。**读取契约与评审侧一致（v0.7.0 钉死）：只读该目录根下的 `.md`、不递归子目录**。这里的规则独立于 `config.yaml` pack 开关，直接全量按 `dimension` + `applies_to` 加载作设计约束；读不到（目录不存在）就跳过。设计侧与评审侧读同一份本地规则，**开发-评审标准一致、单一源不漂移**。
- **v1 只消费，不改 schema**：designer/judge 读取已启用规则包；`dimension` 不匹配设计场景时按规则的 `summary` 字段摘要注入。
- **`applies_phase` 字段推迟到 v2**：v1 不引入「规则适用于 design 还是 review 阶段」的字段，也不在评审侧加 phase 过滤（那会牵动 `rules/README.md` 消费逻辑、`branch-review-guard/SKILL.md` 规则机制、各 `prompts/review-*.md`、`commands/rule.md`、`prompts/add-rule.md` 等十余处，不在 v1 范围）。design 与 review 共享 rules 供给是**单向**的：rules → design 消费；**design 不向 rules 产出**（设计裁决不沉淀为规则，规则只来自评审侧 distill/rule）。

## 调用方式

任意能读本文件的 Agent 都可按下面流程执行。建议提示语（裸读形态）：

- `读取 tools/design-panel/SKILL.md，对 <需求> 做多视角设计方案对比`

> canonical 布局是 `tools/<skill名>/`，**无嵌套形态**——不得写 `tools/branch-review-guard/design-panel/`（该路径永远不会存在）。

命令（**Claude Code 插件形态**为命名空间命令 `/branch-review-guard:design`；其它形态用 `/branch-review-guard design` 或自然语言触发）：

```
/branch-review-guard:design <需求描述>
    [--input <需求/PRD 文档路径>]
    [--module <模块路径前缀>]
    [--variants <2..5>]
    [--quick | --thorough]
```

- 第一段自由文本 = **需求描述**（按优先级：① `$ARGUMENTS`；② `--input` 文档；③ 都缺时自动扫仓库根与各模块 `docs/` 下的 `*_DESIGN.md`/`*_REQUIREMENT.md`/`*_CONTRACT.md` + `git log` 推断；全部落空才回问——见 `## 需求模糊处理`）。
- `--module`：限定勘察范围（如 `modules/order/`），既提高相关性又是成本闸门。
- `--variants`：视角数，缺省按档位（quick=2 / 默认=3 / thorough=5）。
- `--quick` / `--thorough`：成本档位（见 `## 成本档位与规模降档`）。

## 工作流程

> **自主一气呵成，中途不停**：被触发后，**自动连贯执行下面 P0→P6 全部阶段、直接产出完整设计报告**，不要在任何中间阶段停下来向用户提问或征求「是否继续 / 是否开始」。**允许中止并回问用户的情形**仅限 L2 逃生阀（承重 ∧ P1 勘察后仍不可推断 ∧ 二元翻转推荐；或目标行为本身缺失/自相矛盾），恰好一次的终止性集中回问，见 `## 需求模糊处理`。除此之外——**包括需求很大、读码范围很广**——都按流程自动跑完，绝不因「先确认一下」而暂停。特别地，**P1.5「假设预检段」是打印给用户看、同一回合直接续跑的非阻塞动作，不是回问节点**。下列 P0–P6 都是你**自己执行**的内部阶段，不是与用户的交互节点。每阶段开始时向用户打印一行进度。

- **P0 定标**：解析参数；做「需求充分性自检」（目标行为 / 涉及业务域 / 硬约束 / 验收判据 / 非目标）；缺项先自己补（读代码、读文档、读 commit），**能推断的绝不问**——推断结果全部进报告开头的「假设清单」（每条标推断依据）。规模预判决定档位与是否降档（见 `## 成本档位与规模降档`）。产出「需求卡」。打印进度 `[0/6] 需求定标完成：N 视角 X 档，预计 Y 分钟`。
- **P1 建事实底座**：编排器（或派 1 个 Explore）勘察相关模块——入口、数据模型、**现有分层与命名约定**、已启用 `rules/` 栈包相关规则摘要。产出《代码事实包》：**只含事实与约束，零方案倾向**（**不得**含「现有类似实现」这类带方案倾向的筛选——否则 N 个代理共享同一解法锚点、诱发趋同，自毁 P2 隔离独立性）。打印进度 `[1/6] 事实勘察完成：涉及 X 模块 Y 文件`。
- **P1.5 假设预检段（主路径 L1，非阻塞）**：派 designer 前，把「承重假设 Top-N」（N≤5，每条含推断依据 file:line + 若错波及哪个方向）打印给用户看一眼，**编排器不 yield、同一回合直接续跑**（用户可 Esc 打断纠偏——等同现状「进度行 + Esc 中止」，只是 payload 升级为含承重假设）。同时判定 L2 逃生阀：逐条查是否满足「承重 ∧ P1 勘察后仍不可推断 ∧ 二元翻转推荐」，命中（通常 0 条）才升级为一次终止性回问，否则不停进 P2。详见 `## 需求模糊处理` 与 `prompts/orchestrate-design-panel.md`。打印进度 `[1.5/6] 承重假设已列（N 条），继续派 designer…`。
- **P2 独立成案（并行 N）**：派 N 个 `dsp-designer`，各持一个价值取向 + 需求卡 + 事实包；**互不知道彼此存在**（prompt 不提「还有其它方案」）。每人产出一份「方案卡」（含承重论断 ≤4 条，每条带 file:line 证据）。打印进度 `[2/6] N 个设计代理并行出方案中…`。
- **P3 对抗质证**：每个方案派 1 个 `dsp-skeptic`，集中反驳该方案全部承重论断；有效反驳必须给 file:line 级反证。产出每条论断的裁决（维持 ✓ / 削弱 ⚠ / 推翻 ✗）。打印进度 `[3/6] 质证完成：A x/x，B x/x，C x/x`。
- **P4 裁判裁决**：派 1 个 `dsp-judge`：只依据「质证后论断状态 + 方案卡 + 事实包」打分，**不重新探索代码**；产出对比表、推荐主方案、从落选方案嫁接亮点的综合方案（每处嫁接标注来源与依据）。打印进度 `[4/6] 裁决完成：推荐 X（嫁接 …）`。
- **P5 诚实边界核查**：**编排器**（非 judge，避免自打分自核查）跑固定 checklist 定稿前核查（见 `## 诚实边界核查`）。打印进度 `[5/6] 诚实边界核查通过`。
- **P6 产出报告与回复**：按 `templates/design-report-template.md` 落两份文件（完整报告 + 精炼设计稿）；回复 = 链接 + 结论先行 + 完整正文 + 衔接文案。打印进度 `[6/6] 报告已落 …/reports/`。

## 需求模糊处理（三层：优先不打断，极少数才停）

设计阶段的输入（需求）天然模糊，与评审侧（diff 确定）不同：开发者常一轮讲不清需求、细节留有不确定点。若编排器直接猜一个答案就开跑整条擂台，猜错时用户要等整轮（7 子代理、~50–120 万 token、8–15 分钟）跑完、在报告里才看到假设、再返工——本可开跑前一句话解决的，变成跑完一整轮再重来。

因此分三层，**优先「不打断」、把「停下来问」留给极少数关键岔路口**（详细流程见 `prompts/clarify-requirement.md` 与 `prompts/orchestrate-design-panel.md` 的 P1.5）：

1. **L1 开跑前亮假设（主路径，每次都做，不停）**：P1 建完事实底座、P2 派 designer 前，把「承重假设 Top-N」打印给用户看一眼（每条含推断依据 + 若错波及哪个方向），**编排器不 yield、同一回合直接续跑**。用户若觉得不对可主动 Esc 打断纠偏——等同现状「进度行 + Esc 中止改档」，只是把 payload 从「档位/时长」升级为「档位/时长 + 承重假设」，是同一非阻塞点的信息超集，**不比现状更容易卡住**。这让用户第一次能在花掉 50–120 万 token 前看到编排器假设了什么。
2. **L2 逃生阀回问（罕见，才停）**：仅当某承重假设**同时**满足「① 承重（改变方案走向）② P1 勘察后仍不可推断 ③ 猜错会让推荐方案整个反掉」三条时，才升级为**恰好一次**的**终止性**集中回问（≤3 问、一条消息、选择题）。判定放在 **P1 勘察之后**（P0 阶段代码没系统读，此时说「查不出来」往往是「还没查」，会误触发——这是 P0 信息不全的老坑）。目标行为本身缺失/自相矛盾时同样走 L2，且无需等 P1。
3. **L3 兜底（不停）**：其余一切细节不确定 → 进「假设清单」写进报告开头（每条标推断依据），事后纠偏。**能推断的绝不问**。

⚠️ **为什么不「有不确定就停下来问」+ 不得承诺「不回复则按默认继续」**：本仓库是纯 Markdown + Git，无 hook/超时机制；agent 一旦提问即结束本轮，用户不回就永远卡住、只能从头重来——这正是 v0.2.9 修掉的「停在等用户输入」bug。关键不对称：现状用户主动按 Esc 是「默认继续、静默安全」，而 agent 主动发问是「默认死局、静默不安全」。故 L1 优先（不停、只亮信息），L2 停顿只留给「不问就整轮白跑」的关键岔路口。

## 子代理设计

新增 3 个只读子代理，前缀 `dsp-`（design-panel 缩写），与评审侧 `bru-` 对称区隔。**沿用 `bru-skeptic` 的「1 个定义文件 × 编排器注入视角 × N 实例」先例**——视角定义集中放在 `prompts/design-architect.md`，新增视角只改 prompt 不加 agent 文件。

- **`dsp-designer`**（`agents/dsp-designer.md`，按视角派 N 实例）：输入需求卡 + 事实包 + 指定视角 + rules 摘要；产出方案卡（摘要 / 改动点清单 / **承重论断 ≤4 条带 file:line** / 风险与回滚 / 工作量 / 需运行时验证项）。视角池：最小改动 / 长期可维护 / 风险与回滚（默认档前 3）+ 性能与容量 / 复用与删减（彻底档追加）。
- **`dsp-skeptic`**（`agents/dsp-skeptic.md`，每方案 1 个实例）：与 `bru-skeptic` 同族但**不复用其文件**（输入契约是发现格式、「规则校准」视角对设计论断无意义，硬套污染两侧）；复制其有效反驳标准，分离文件。每条论断从三角度轮询（事实性 / 完备性 / 代价），一人全担。
- **`dsp-judge`**（`agents/dsp-judge.md`，定稿前 1 个）：打分（需求契合度 30% / 改动面与实施风险 25% / 长期可维护性 20% / 可测试与可验证性 15% / 工作量性价比 10%，诚实度为扣分项）+ 嫁接硬产出 + 同质坦白。

## 对抗质证护栏（四件套）

> 公共缺口补救：本 skill 砍掉了评审侧 `verify-findings.md` 的 3 票聚合（≥2 票才杀），单挑战者误判会腐蚀横向排名；而「缺反证按维持计」只防误杀（假推翻）、**不防假维持**（挑战者偷懒全判维持）。擂台是横向比较，各方案挑战者严格度方差直接改写排名。落地四道护栏（写进 `dsp-skeptic` 与 `dsp-judge`）：

1. **核查动作记录**：挑战者随裁决附「实际读过哪些 file:line」的取证记录，供裁判审计；空取证的全维持裁决视作偷懒。
2. **论断外硬伤出口**：除逐条质证承重论断，怀疑者**必须**提报「论断清单之外的、方案与仓库事实直接冲突的硬伤」——坏方案的典型死因是遗漏既有机制，自报论断可全为真，只攻自报论断会漏网。
3. **推翻票裁判抽验**：「推翻」票由裁判**亲自抽验反证 file:line** 后才生效计入——防单挑战者把死代码当生效逻辑造成的假推翻翻转推荐。
4. **战绩不进权重**：「质证战绩」（维持/削弱/推翻条数）只作展示行，**不单独作为打分权重项**——排名依据裁判综合维度，而非挑战者抽签运气。

## 诚实边界核查（P5，定稿前必经）

> 设计文档虽无评审的「覆盖率声明」，但 `bru-critic` 的诚实边界核查项在本 skill 有明确靶子：本 skill 明令禁止「性能无影响」类已验证式表述、报告设「需运行时验证项」专章——「设计代理是否违反这条禁令」正是可对账对象。砍掉独立核查 = 诚实边界成无人看守的承诺。

落地：**不新增独立 critic 子代理**（控成本），P5 由**编排器**（非 judge）跑固定 checklist：

- 扫描报告有无「已验证通过 / 性能无影响」类违规表述；
- 核对嫁接项均有来源标注（取自方案 X 的 Y）；
- 核对需求卡要点是否被推荐方案全覆盖；
- 核对降级模式声明是否如实落章（见 `## 降级策略`）；
- 核对「需运行时验证项」专章是否齐全。

即 `bru-critic` 的设计侧精简版——一个 checklist 步骤而非新子代理。

## 降级策略

- **不支持子代理的环境**：顺序逐视角出方案。**诚实声明独立性弱化**——顺序模式下编排器写完方案 A 后 A 已在其上下文里，「落盘暂存、禁止回看」是**伪机制**、无法消除上下文残留，P2 的隔离收益为零。
  - ⚠️ **两套口径刻意不同，不得混用**：评审侧降级口径是「结果一致，只是更慢」（`branch-review-guard/SKILL.md` 可移植性节）；**设计侧用「独立性弱化」**——设计侧的隔离即产品，同上下文顺序生成会趋同，不能套用「结果一致」。报告必须如实标注「本次为顺序执行，方案独立性弱于并行隔离模式，结论仅供参考」。
  - 质证与裁决同样退化为编排器戴帽自查。
- **子代理失败/超时**：designer 失败 → 重派一次；再失败 → 降 N-1 并声明（只剩 1 个则退化为「单方案 + 质证」模式）。skeptic 失败 → 该方案论断标「未质证」，裁判对未质证论断按七折计分并标注。judge 失败 → 编排器按同一 prompt 自任裁判并声明。
- **方案卡质量门**（编排器收卡时机械校验）：方案卡缺「承重论断」或论断无 file:line 证据 → 打回重派一次（附具体缺项）；仍不合格 → 弃用该方案并声明。挑战者反驳缺 file:line 反证 → 按维持计。

## 成本档位与规模降档

| 档位 | designer | skeptic | judge | 子代理总数 | token 量级（估） | 时长（估） |
|---|---|---|---|---|---|---|
| `--quick` | 2 | 0（编排器抽查论断证据） | 编排器自任 | 2 | ~20–50 万 | 3–6 分钟 |
| **默认（标准）** | 3 | 3（每方案 1） | 1 | 7 | ~50–120 万 | 8–15 分钟 |
| `--thorough` | 5 | 5（每方案 1，三角度更深） | 1 | 11 | ~120–250 万 | 15–30 分钟 |

> 量级为**估算**，随仓库与需求规模浮动；designer 被要求每条论断附 file:line 证据、必须亲自读码取证，勘察成本不可压到极低。进度行先报本次档位与预计时长，用户可 Esc 中止改档。

**规模预判自动降档**（P0 判定是否降档为单方案直答，1-2 分钟）：

- **降档触发（合取）**：预估改动 ≤5 文件 **且** 无下述一票否决项 **且** 需求文本未指定唯一做法。
- **一票否决降档**（即使文件数 ≤5 也必须跑擂台，因这些恰是最需要多方案的高风险小 diff）：涉及对外契约变更（HTTP API / RPC / 消息队列 / 对接协议）、存储结构变更（DB 迁移、表/字段/索引）、删除/类型/枚举/必填字段变更（对齐评审侧 `branch-review-guard/SKILL.md` 高风险全量深读清单）、有跨模块权衡。
- 用户显式传 `--variants N` 时，**显式意图优先**、不降档。
- 降档后报告开头声明「本需求解法空间小，已降档为单方案直答」——不回问、不硬跑擂台。

## 输出物

**报告落地路径**（复用评审侧双路径逻辑）：优先 `tools/branch-review-guard/reports/`（安装器路径已建则用之，已被该 skill 条目 `reports_gitignore` 覆盖），否则项目根 `branch-review-reports/`（不存在即创建，建议加入 `.gitignore`）。

- **完整报告**：`design-panel-<需求slug>-<shortSha>-<timestamp>.md`，靠前缀与评审报告区分。结构见 `templates/design-report-template.md`（结论先行 / 需求卡 / 事实包摘要 / 方案详情+质证裁决 / 裁决细节与嫁接 / 推荐方案落地路线 / 需运行时验证项 / 附录被驳论断与反证）。
- **精炼设计稿**：同目录另出 `<需求slug>_DESIGN.md`（只含推荐方案 + 嫁接后最终形态 + 落地路线，~1 页）。回复中提示用户认可后移入项目 `docs/` 提交——`branch-review-guard` 评审的「建立上下文」步骤会自动读 `docs/*_DESIGN.md`，**设计→评审闭环**成立。

## 规则弹性路径解析

designer/judge 读取 `rules/` 时，按以下顺序查找**首个存在**的规则目录（与评审侧一致）：

0. **作为 Claude Code 插件运行时（优先）**：`rules/` 随 `branch-review-guard` 一同加载，直接读取，**无需按文件路径查找**。
1. `tools/branch-review-guard/rules/`（canonical 正本，本 skill `requires` branch-review-guard，该目录随之就位）；
2. 项目根 `rules/`（裸读形态可能的位置）。

任一命中则读取并按 `## 规则机制` 消费；全部找不到则**缺包降级**——通用设计视角照常跑，报告「已启用规则包」注明「未就位、栈特有约束未注入」，**不伪造**。

> 此外，**项目本地 `branch-review-rules/`**（被设计项目根）始终 **best-effort 叠加**读取——它与上述插件 rules **并列叠加**、不是二选一；详见 `## 规则机制` 的「项目本地规则叠加」。

## 回复约定

最终回复必须**同时**包含报告链接和报告正文（先链接、结论先行、再完整正文）。所有面向用户的输出用中文（代码标识符、命令、文件路径、类名、方法名保留原文）。正文后固定附衔接文案：

```markdown
报告文件：[<filename>](<relative-path>)

# <需求名> 设计方案对比报告

...完整报告正文...

---
是否需要我把推荐方案展开成文件级开发 todo 清单（按依赖顺序排列，每项标注涉及文件与验收点）？开发过程中可随时用 /branch-review-guard:diff 边写边查，提测前用 /branch-review-guard:review 整分支评审。若认可精炼设计稿（<slug>_DESIGN.md），建议移入 docs/ 提交——后续评审建立上下文时会自动读到它。
```

报告很长时也必须保留结论先行与对比表；必要时只压缩较长的方案卡原文。衔接文案为**固定文案**，每次设计回复末尾都要附上。

## 可靠性规则

- 方案与论断可定位（`file:line`）；区分**事实 / 推测 / 待运行时验证**。
- 不编造调用方与现有结构；设计改动给依据。
- 被反驳的论断支撑的优势**不得计分**；分差 <0.5 时写「两案接近，选择取决于 X」，不硬造赢家。
- 两案改动点重叠 >80% 时如实报告「该需求解法收敛，无真实权衡空间」——这本身是有价值的结论。
- 性能/并发/迁移类论断只给「需运行时验证项」，不下「已通过」。
- 显式声明顺序模式的独立性弱化，**不装作与并行等效**。

## 与 distill / rule 的关系（单向：消费不产出）

- **消费**：designer/judge 读取已启用 `rules/` 规则包作为设计约束与降噪校准。
- **不产出**：设计阶段的裁决与「需运行时验证项」**不回流** `distill`/`rule`——规则只来自评审侧（`distill` 聚类评审报告、`rule` 手动录入）。`distill` 取样时**排除** `design-panel-*` 与 `*_DESIGN.md`（见 `commands/distill.md` 取样前缀过滤），避免设计报告污染评审规则的聚类样本。
