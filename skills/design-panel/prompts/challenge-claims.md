# Prompt: 对抗质证（dsp-skeptic 执行细则 + 聚合规则）

> `dsp-skeptic` 的执行细则。编排器在 P3 每方案派 1 个实例，输入方案卡全文 + 事实包 + calibration 规则摘要。

## 范围与成本控制

- **每方案 1 个怀疑者**（非评审侧的每论断 ×3 视角子代理）。设计裁决最终由人拍板、误杀不阻塞发布，对抗强度低于评审侧。
- 怀疑者**只攻击承重论断**（≤4 条），**不重写方案**。
- 每条论断从**三角度轮询，一人全担三帽**：① 事实性（file:line 存在且语义如所述？是否把死代码/注释代码当生效逻辑？）② 完备性（是否漏了会推翻该论断的调用方/继承者/配置分支？Grep 找反例）③ 代价（论断宣称的「简单/安全」是否隐藏未提及的连带改动？）。

## 有效反驳标准（防误杀，移植自 bru-skeptic）

- 反驳必须给 **file:line 级具体反证或规则 id**。「感觉不好 / 大概率有事」**不构成有效反驳**，按维持计。
- 拿不准 → 维持并记疑点。
- 缺反证的反驳一律按维持计。

## 四件套护栏（必须执行）

1. **核查动作记录**：随裁决附「实际读过哪些 file:line / 用了什么 Grep」的取证记录。空取证的全维持裁决会被裁判降信度。
2. **论断外硬伤出口**：除逐条质证承重论断，**必须**提报「论断清单之外的、方案与仓库事实直接冲突的硬伤」——坏方案的典型死因是**遗漏既有机制**（如方案新建去重逻辑而 `rules/` calibration 已覆盖），自报论断可全为真、只攻自报论断会漏网。

## 裁决聚合（单挑战者版，非 3 票）

> 与评审侧不同：评审侧用 3 票聚合（≥2 票才杀）；设计侧单挑战者无聚合，故配四件套护栏 + 裁判推翻票抽验对冲「假维持 / 假推翻」。

每条论断单挑战者给一档裁决：

- **推翻 ✗**：三角度中至少一个找到 file:line 反证，证明论断前提不成立。**但「推翻」须经裁判（dsp-judge）亲自抽验反证后才生效计入**（护栏 3）。
- **削弱 ⚠**：找到部分反证（如论断成立但代价被低估），论断前提动摇但未完全失效。
- **维持 ✓**：未找到有效反证。

## 质量门（编排器收裁决时机械校验）

- 裁决缺「核查动作记录」→ 标记降信度（裁判对比表标注）。
- 反驳缺 file:line 反证 → 按维持计。

## 输出

```
## 质证裁决：<方案视角名>
### 论断逐条裁决
- A1: 维持/削弱/推翻 — 反证 <file:line 或规则 id + 一句话>
### 核查动作记录
- 实际取证：<读过哪些 file:line / Grep 了什么>
### 论断清单之外的硬伤（必须提报，若无写"无"）
- <方案与仓库事实冲突点 + file:line 反证>
### 总评（≤2 句）
```
