# design-panel 设计文档

> **本文件是什么**：这是 design-panel **技能本身**的设计文档（存于插件仓库），不是运行时产出。区别——运行时 design-panel 对某个具体需求产出的设计稿是目标项目里的 `<需求slug>_DESIGN.md`，落报告目录、提示用户认可后移入 `docs/` 供 `:review` 的"建立上下文"读取（`skills/branch-review-guard/SKILL.md:67` 已支持读 `docs/*_DESIGN.md`）。两者同名易混，本文统一称前者为「本设计文档」、后者为「运行时设计稿」。

## 0. 一句话定位

**design-panel** = 需求方案设计阶段的多视角设计编排器：并行派 N 个互相隔离的设计子代理从不同价值取向独立读代码成案 → 对每案的承重论断派怀疑者做 file:line 级对抗质证 → 裁判打分产出「对比表 + 推荐方案 + 嫁接落选亮点的综合方案」单份中文设计文档。与 `branch-review-guard` 形成「设计侧出方案 / 评审侧守出口」的双子星。

命令：`commands/design.md` → `/branch-review-guard:design`（遵循仓库 `<verb>.md` 命名惯例）。

## 1. 这份文档怎么来的（过程留证）

本设计**不是单 agent 一把梭**，而是用 design-panel 自身要实现的「设计擂台」模式跑出来的，留证如下：

- **设计代理 ×3**（互相隔离、互不可见）：A 最小可用优先 / B 架构对称优先 / C 体验与成本优先。
- **怀疑者 ×3**（每方案 1 个，只攻该方案的承重论断，要求 file:line 级反证）。
- **裁判 ×1**：拿「三方案 + 三份质证裁决」综合打分、嫁接。

### 1.1 对比打分表

| 维度 | A 最小可用 | B 架构对称 | C 体验成本 |
|---|---|---|---|
| 1. 需求满足度 | 4 | **5** | 4 |
| 2. 论断存活率与事实可靠性 | 2（A3 被推翻 + 2 处硬伤） | **3.5**（B1/B2 骨架论断维持） | 3 |
| 3. 落地成本与工作量诚实度 | **4** | 2（宣称改 10 处实际 ≥14-15 处） | 3 |
| 4. 与仓库架构及纪律一致性 | 2（与 manifest 纪律正面冲突） | **4** | 3（澄清关卡踩 0.2.9 同型 bug） |
| 5. 使用体验 | 3 | 3 | **4.5** |
| 6. 长期可维护性与演进空间 | 3 | **4** | 3.5 |
| **总分** | **18/30（3.0）** | **21.5/30（3.58）** | **21/30（3.5）** |

### 1.2 推荐

**B 与 C 总分差 0.08，属实接近；骨架推荐 B。** 理由：三案中唯有 B 的两条骨架级论断（B1 归属现有插件、B2 单文件参数化设计代理）双双质证维持，其全部失分集中在可剥离替换的集成细节（applies_phase、distill glob）；而 C 的骨架相邻论断四条全部削弱，其真正的强项（体验层）全是可嫁接模块。**最终形态 = B 的结构骨架，剥掉被证伪的 rules/distill 集成规格，大面积嫁接 C 的体验层与 A 的成本纪律**，并吸收全部质证暴露的公共缺口修补。

### 1.3 名称裁决

**design-panel**（不用 C 提议的 arena）。本机制中方案与方案**从不见面**，对抗发生在「怀疑者 vs 论断」、裁决出自评委——这正是评委团/合议庭（panel）而非擂台（arena）的结构；且与工作名一致、与 B 落地清单一致，改名零收益。

## 2. 最终形态（综合方案）

### 2.1 归属与形态

- **归属现有插件**（B1，质证维持）：放进 `branch-review-guard` 插件做双子星，不新建插件——共享 `rules/`、`distill`/`rule` 闭环、安装器、版本纪律与 marketplace 声明。
- canonical 源目录：`skills/design-panel/`。
- 子代理前缀：`dsp-`（与评审侧 `bru-` 对称区隔）。

### 2.2 触发方式与输入

```
/branch-review-guard:design <需求描述>
    [--input <需求/PRD 文档路径>]
    [--module <模块路径前缀>]
    [--variants <2..5，默认按档位>]
    [--quick | --thorough]
```

**需求描述来源**（按优先级）：① `$ARGUMENTS` 自由文本；② `--input` 文档；③ 都缺时扫仓库根与各模块 `docs/` 下的 `*_DESIGN.md`/`*_REQUIREMENT.md`/`*_CONTRACT.md` + `git log` 推断。

**需求模糊处理**（吸收 C3 修补 + A 的成本纪律）：
- 触发后第一步做「需求充分性自检」（目标行为 / 涉及业务域 / 硬约束 / 验收判据 / 非目标）。缺项**先自己补**——读代码、读文档、读 commit。
- **能推断的绝不问**——推断结果全部进运行时设计稿开头的「假设清单」（每条标推断依据），让用户事后纠偏。**这是唯一可靠的兜底**（质证认定）。
- 仅当**目标行为本身**缺失或自相矛盾时，允许**恰好一次**的**终止性**集中回问（≤3 问、一条消息问齐）。此后无论如何不再停。
- ⚠️ **删除一切「不回复则按默认继续」类承诺**（C3 削弱 + 硬伤 1）：本仓库是纯 Markdown + Git，**无 hook/超时机制**，agent 一旦提问即结束本轮；「不回复则默认续跑」不可实现，正是 `CHANGELOG.md` 0.2.9 修掉的「停在等用户输入」同型 bug。

### 2.3 工作流程（P0–P6）

> **自主一气呵成**（除 §2.2 的终止性单次回问外）：触发后连贯执行 P0→P6 直接产出报告，不停下征求「是否继续」。每阶段开始向用户打印一行进度（吸收 C）。该措辞的载体文件见 §3.2 的冗余清单。

| 阶段 | 做什么 | 产出 | 进度行示例 |
|---|---|---|---|
| **P0 定标** | 解析参数；需求充分性自检（可能终止性单次回问）；**规模预判**决定档位与是否降档（见 §2.8） | 需求卡（目标/约束/假设清单/非目标） | `[0/6] 需求定标完成：3 视角标准档，预计 8-15 分钟` |
| **P1 建事实底座** | 编排器（或派 1 个 Explore）勘察相关模块：入口、数据模型、**现有分层与命名约定**、已启用 `rules/` 栈包相关规则摘要 | 《代码事实包》（**只含事实与约束，零方案倾向**；详见 §2.4 的中性化纪律，吸收 C2 修补） | `[1/6] 事实勘察完成：涉及 X 模块 Y 文件` |
| **P2 独立成案**（并行 N） | 派 N 个 `dsp-designer`，各持一个价值取向 + 需求卡 + 事实包；**互不知道彼此存在** | 每人一份「方案卡」（见 §2.4） | `[2/6] 3 个设计代理并行出方案中…` |
| **P3 对抗质证** | 每方案派 1 个 `dsp-skeptic` 集中反驳其全部承重论断（≤4 条）；有效反驳必须 file:line 级反证 | 每条论断裁决：维持 ✓ / 削弱 ⚠ / 推翻 ✗（附反证） | `[3/6] 质证完成：A 4/4，B 3/4，C 2/4` |
| **P4 裁判裁决** | 派 1 个 `dsp-judge`：只依据「质证后论断状态 + 方案卡 + 事实包」打分，**不重新探索代码**；产出对比表、推荐、嫁接综合方案 | 打分表 + 推荐 + 嫁接清单 | `[4/6] 裁决完成：推荐 B（嫁接 A 的迁移策略）` |
| **P5 诚实边界核查**（吸收 A3 推翻的修补） | **编排器**（非 judge）跑固定 checklist 定稿前核查（见 §2.5） | 定稿 | `[5/6] 诚实边界核查通过` |
| **P6 产出报告与回复** | 落完整报告 + 精炼运行时设计稿；回复 = 链接 + 结论先行 + 完整正文 + 衔接文案 | 两份文件 | `[6/6] 报告已落 …/reports/` |

**回复末尾固定衔接文案**：todo 展开提示 + 「开发中用 `:diff` 边写边查、提测前 `:review` 整分支评审」+ 「认可运行时设计稿则建议移入 `docs/` 提交——后续评审建立上下文会自动读到」。

### 2.4 子代理设计（3 个 `dsp-*`，单文件参数化视角）

沿用 `bru-skeptic` 的「1 个定义文件 × 编排器注入视角 × N 实例」先例（B2，质证维持——`agents/bru-skeptic.md:3,15-18` 已运行该模式）。

**每个 agent 定义文件 frontmatter 必须写 `tools: Read, Grep, Glob, Bash`**（吸收 A1 削弱 + 硬伤 1 的修补）：只读必须是**机制级工具白名单**，不是 prompt 里的一句话——通用子代理（general-purpose）在本插件环境工具集是 `*`（含 Edit/Write），靠 prompt 写「只读」只是指令纪律，无法在机制上排除「设计代理顺手 Edit 目标项目文件」的事故。

#### `agents/dsp-designer.md`（1 个定义，按视角派 N 实例）

- **工具**：`Read, Grep, Glob, Bash`（只读）；`model: inherit`。
- **输入**：需求卡 + 《代码事实包》+ 指定视角 + 已启用 rules 栈包相关规则摘要（栈特有的坑作为设计约束）。
- **视角池**（默认档前 3 / 彻底档前 5）：

| 视角 | 优化目标 | 默认 | 彻底 |
|---|---|---|---|
| 最小改动优先 | 最小 diff 面、复用现有扩展点、不动稳定代码 | ✅ | ✅ |
| 长期可维护优先 | 正确抽象、还技术债、为下个需求铺路 | ✅ | ✅ |
| 风险与回滚优先 | 灰度/回滚/兼容性、故障半径最小 | ✅ | ✅ |
| 性能与容量优先 | 调用链放大、热点路径、数据量级 | — | ✅ |
| 复用与删减优先 | 现有能力复用最大化、净代码量最小 | — | ✅ |

- **prompt 要点**：
  - 「你看不到其他方案，也不要猜测或迎合其他视角——你的价值在于把**你的取向推到底**，权衡由裁判做。」
  - 产出**结构化方案卡**：方案摘要（≤5 句）→ 改动点清单（文件级：新增/修改/删除 + 每处一句话）→ **承重论断 ≤4 条**（编号 A1…，每条一句话 + **file:line 证据**——这是可被反驳的赌注，如「A1: `OrderService.applyDiscount` 是唯一折扣入口（`OrderService.java:88`）」）→ 风险与回滚 → 工作量估算（S/M/L + 依据）→ 需运行时验证项。
  - **移植诚实边界**（继承 `SKILL.md:17,182`）：性能/容量/并发类断言**禁止**写成「性能无影响」这类已验证式表述，只能进「需运行时验证项」。
  - **成本硬上限**（吸收 A §9）：读码范围受需求卡约束，超范围只允许追依赖链、不允许全仓漫游；承重论断统一硬上限 ≤4 条/方案。
- **《代码事实包》中性化纪律**（吸收 C2 削弱的修补）：事实包**不得**含「现有类似实现」这类带方案倾向的筛选——否则 N 个代理共享同一解法锚点，诱发趋同，自毁 P2 的隔离独立性（C1）。只收录：相关模块入口/数据模型清单、现有分层与命名约定、已启用规则摘要。需要全仓能力盘点的视角（如复用优先）由该 designer **自行**越出事实包 Grep 取证——这是该视角的职责，不是事实包的职责。

#### `agents/dsp-skeptic.md`（每方案 1 个实例）

- **工具**：`Read, Grep, Glob, Bash`；`model: inherit`。
- 与 `bru-skeptic` 同族但**不复用其文件**（A2，质证维持——`bru-skeptic` 输入契约是发现格式 `[P*] 维度—问题—file:line`、且「规则校准」视角对设计论断无意义，硬套污染两侧语义）。**复制其有效反驳标准，分离文件**。
- **输入**：一份完整方案卡 + 事实包 + 已启用 calibration 规则摘要。**只攻击承重论断，不重写方案**。
- 每条论断从三角度轮询（**一人全担三帽**，非 review 的每论断 ×3 子代理——见 §2.5 护栏与 §2.8 成本）：① 事实性（引用 file:line 存在且语义如所述？）② 完备性（是否漏了会推翻该论断的调用方/继承者/配置分支？Grep 找反例）③ 代价（论断宣称的「简单/安全」是否隐藏未提及的连带改动？）。
- **有效反驳标准**（移植 `bru-skeptic.md:20-24`）：反驳票必须给 **file:line 级具体反证或规则 id**；「感觉不好」按维持计；拿不准 → 维持并记疑点；缺反证的反驳一律按维持计。
- **输出**：每条论断一行 `A1: 维持 / 削弱(附反证) / 推翻(附反证)` + 备注。

#### `agents/dsp-judge.md`（定稿前 1 个）

- **工具**：`Read, Grep, Glob, Bash`；`model: inherit`。
- **输入**：需求卡 + 全部方案卡 + 全部质证裁决 + 事实包。**不重新探索代码**（争议已由怀疑者用代码裁过；省成本）。
- **打分维度与权重**：需求契合度 30% / 改动面与实施风险 25% / 长期可维护性 20% / 可测试与可验证性 15% / 工作量性价比 10%。另设**诚实度**为扣分项（把「需验证」写成「已验证」直接扣分）。
- **裁决纪律**（吸收 C §4.3）：被「推翻」的论断支撑的优势**不得计分**；「需运行时验证项」不得作为决定性加分；分差 <0.5 时**必须**写「两案接近，选择取决于 X」而非硬造赢家；两案改动点重叠 >80% 时如实报告「该需求解法收敛，无真实权衡空间」。
- **嫁接是硬产出**：必须逐个检视落选方案的「维持 ✓ 论断」，能移植进推荐方案的列为嫁接项（含移植方式），不能的写明原因。
- **兼任 critic 的设计侧变体**（吸收 B §4 + A3 修补）：核对「每方案是否真覆盖需求全部要点、有无视角实质趋同、嫁接是否都有来源标注」。

### 2.5 质证护栏四件套（核心修补——三方案公共缺口的补救）

> 三方案的质证共同暴露一个公共缺口：都砍掉了评审侧 `prompts/verify-findings.md:20-25` 的 **3 票聚合防线**（≥2 票才杀），单挑战者误判会直接改写横向排名，而「缺反证按维持计」只防误杀（假推翻）、**完全不防假维持**（挑战者偷懒全判维持）。设计擂台是横向比较，各方案挑战者严格度方差直接腐蚀排名公平性。

落地为四道护栏，写进 `dsp-skeptic` 与 `dsp-judge` 两份 prompt：

1. **核查动作记录**：挑战者必须随裁决附「实际读过哪些 file:line」的取证记录，供裁判审计——空取证的全维持裁决可视作偷懒。
2. **论断外硬伤出口**：除逐条质证承重论断外，怀疑者**必须**提报「论断清单之外的、方案与仓库事实直接冲突的硬伤」（吸收 A5 削弱的修补——坏方案的典型死因是遗漏既有机制，自报论断可全为真，质证只攻自报论断会漏网）。
3. **推翻票裁判抽验**：「推翻」票由裁判（`dsp-judge`）**亲自抽验反证 file:line** 后才生效计入——防单挑战者把死代码当生效逻辑（`bru-skeptic.md:16` 提示的易错点）造成的假推翻翻转推荐。
4. **战绩不进权重**：「质证战绩」（维持/削弱/推翻条数）只作展示行，**不单独作为打分权重项**——排名依据裁判综合维度，而非挑战者抽签运气。

### 2.6 诚实边界核查（定稿前必经步骤——A3 推翻的修补）

> A3「设计文档无覆盖率声明、critic 无靶子」被**推翻**：`bru-critic.md:23` 核查清单第 5 条「诚实边界：运行时维度有没有被写成『已验证通过』」在设计产出物里有明确靶子——本技能 §2.4 明令禁止「性能无影响」类已验证式表述、报告设「需运行时验证项」专章，「设计代理是否违反这条禁令」正是可对账对象。砍掉独立核查 = 诚实边界成无人看守的承诺，违反 CLAUDE.md「诚实边界是产品核心卖点、改 prompt 时不要削弱」。

落地：**不新增独立 critic 子代理**（控成本），改为 P5 阶段由**编排器**（非 judge，避免自打分自核查）跑固定 checklist：

- 扫描报告有无「已验证通过 / 性能无影响」类违规表述；
- 核对嫁接项均有来源标注（取自方案 X 的 Y）；
- 核对需求卡要点是否被推荐方案全覆盖；
- 核对降级模式声明是否如实落章（见 §2.7）；
- 核对运行时验证项专章是否齐全。

即 `bru-critic` 的设计侧精简版——一个 checklist 步骤而非新子代理。

### 2.7 降级策略

- **不支持子代理的环境**（对齐 `SKILL.md:23-24`）：顺序逐视角出方案。**诚实声明独立性弱化**（吸收 A §6 是三案中最诚实的表述）：顺序模式下编排器写完方案 A 后 A 已在其上下文里，「落盘暂存、禁止回看」是**伪机制**、无法消除上下文残留，P2 的隔离收益为零。
  - ⚠️ **措辞二选一**（吸收 C 硬伤 3）：评审侧降级口径是「结果一致，只是更慢」（`SKILL.md:24`、`verify-findings.md:13`）；但**设计侧的隔离即产品**（同上下文顺序生成会趋同），不能套用「结果一致」。运行时设计稿必须如实标注「本次为顺序执行，方案独立性弱于并行隔离模式，结论仅供参考」——并在本设计文档登记：**评审侧用「结果一致只是更慢」、设计侧用「独立性弱化」**，两套口径刻意不同，落地时不得混用。
  - 质证与裁决同样退化为编排器戴帽自查。
- **子代理失败/超时**（吸收 C §6）：designer 失败 → 重派一次；再失败 → 降 N-1 并声明（只剩 1 个则退化为「单方案 + 质证」模式）。skeptic 失败 → 该方案论断标「未质证」，裁判对未质证论断按七折计分并标注。judge 失败 → 编排器按同一 prompt 自任裁判并声明。
- **方案卡质量门**（吸收 C §6）：编排器收卡时机械校验——方案卡缺「承重论断」或论断无 file:line 证据 → 打回重派一次（附具体缺项）；仍不合格 → 弃用该方案并声明。挑战者反驳缺 file:line 反证 → 按维持计。

### 2.8 成本档位与规模降档

| 档位 | designer | skeptic | judge | 子代理总数 | token 量级（估） | 时长（估） |

- 涉及**存储结构**变更（DB 迁移、表/字段/索引）；

- 有跨模块权衡。
