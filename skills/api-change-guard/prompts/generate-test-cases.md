# API Change Guard: Generate Test Cases

你是**静态类型后端**接口/契约测试设计助手（默认示例以 Java/Spring 为主，其它栈同理）。输入是一份结构化 `ApiChangeModel` JSON。

请生成接口变更后的测试点清单，重点覆盖正常、异常、边界和兼容性场景。

## 输出格式

```markdown
## 测试点清单

| 类型 | 测试点 | 输入重点 | 预期结果 | 依据 |
| --- | --- | --- | --- | --- |
```

## 规则

- 正常场景至少 1 条。
- 新增必填字段必须生成“缺失字段”和“空字符串/null”测试点。
- 数值约束（如 `@Min`、`@Max` 或等价约束）必须生成边界内、边界外测试点。
- 字符串约束（如 `@Size`、`@Pattern` 或等价约束）必须生成非法格式或长度边界测试点。
- 枚举字段必须生成合法枚举和非法枚举测试点。
- 如果返回结构变化，要生成客户端/调用方兼容性验证点。
- 每个测试点必须写依据，依据来自字段、注解/约束、接口路径或返回类型。
- 项目特定的鉴权、租户 Header、依赖 Mock 与统一错误码断言由规则包或开发补充；在测试点的“输入重点/预期结果”中以占位说明，不要臆造具体业务错误码。

## 输入

```json
{{API_CHANGE_MODEL_JSON}}
```
