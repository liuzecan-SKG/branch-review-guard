# Changelog

本文件记录 Endpoint Performance Review 的版本变更，版本遵循语义化版本（SemVer）。

## [0.4.0] - 2026-07-10

### 新增
- **「修法阶梯 L0/L1/L2」新小节（checklist 后）**：把"尽量不改既有共享代码"从提醒升级为**排序规则**——L0 消费侧/读路径（完全不碰共享文件）> L1 共享增量（只新增方法、不改既有签名/行为）> L2 改既有共享（最后手段）。每条 P0/P1 修法必须先给 L0；L0 明显救不了瓶颈（如常态冷路径"每日首次必冷"）才上 L1；推荐 L2 须额外写明「L0/L1 为何都不行」。明确**阶梯是优先序不是禁令**：窄消费复用大聚合类瓶颈强行 L0 会得到"看着不动共享、其实没解决瓶颈"的方案，证明 L0/L1 不足即可上 L2。
- **报告模板「发现项」加两栏**：每条发现标注「阶梯 L0/L1/L2」+「爆炸半径（受影响调用方数）」；「推荐方案」同步要求逐项标注，推荐 L2 时写明 L0/L1 为何不足——让"动不动共享"在报告里一眼可见。

### 变更
- checklist「测正确」条指向新的「修法阶梯」小节。
- 对抗增强第 3 项（P0 修法质证）追加**质证重心反转**：修法若为 L2，怀疑者追加专攻「动共享必要性」——L2 真不可避免吗？有无 L0/L1 等价方案？须证明是最后手段而非顺手。

### 说明
- 源自用户工程偏好"性能优化方案尽量不修改共享代码"（已同步固化到对抗式编排 cheatsheet 的擂台配方：方案 B 升格为 L0/L1 旗手真候选、rubric 爆炸半径权重 30%）。
- 本次只改 endpoint-perf-review 自身（SKILL frontmatter 0.3.0 → 0.4.0 + checklist + 新小节 + 模板 + 本 CHANGELOG）与 `manifest.json` 该条目版本；`plugin.json` / suite.version 不动（dependency 独立版本，同 0.3.0 惯例）。未增删 skill/agent/命令，计数描述不动。

## [0.3.0] - 2026-07-09

### 新增
- **报告模板新增「需运行时验证项」独立一节**：静态评审无法确认的（真实 QPS/调用量分布、冷/热缓存命中率、EXPLAIN 实际执行计划、锁竞争、降级/熔断真实触发行为、批量改动实际耗时）单列，禁止写成"已验证"——把套件 `rules/baseline/runtime-claims-need-evidence` 的精神直接钉进本 skill 自身模板。原模板只有"风险与护栏"，运行时项易被混入已确认结论。
- **可选「对抗增强」模式（仅 standalone）**：默认单代理一气呵成不变；独立运行且环境支持子代理时，可叠加四层对抗——①调用清单闭合性核查（清单是所有发现的"事实包"，错误会传导，Workflow 第 3 步后派核对代理逆向验字段溯源/成本标注/漏项）②P0/P1 发现派 2 怀疑者分镜头质证（证据核实/触发路径，默认倾向推翻、反证须 file:line，存活项标 `✓已对抗验证`/`⚠有争议`）③P0 修法派怀疑者查"优化引入的新故障"（一致性/并发上下文/共享代码行为变更/爆炸半径）④优先级按 `预期收益 × 实现成本 × 引入风险` 显式 rubric。**被 `branch-review-guard` 第 6 章复用时不启用**（编排器已有 bru-skeptic/critic，避免双重开销）。

### 说明
- 本次只改 endpoint-perf-review 自身（SKILL frontmatter + 模板 + 新小节 + 本 CHANGELOG）与 `manifest.json` 该条目版本（0.2.6 → 0.3.0）；`plugin.json` / suite.version 不动（dependency 独立版本，参见 0.5.0「评审侧及依赖 skill 未改、版本不动」对称惯例）。
- 未增删 skill/agent/命令，故 plugin/marketplace/AGENTS/README 的计数描述不动。

## [0.2.0] - 2026-06-26

### 变更
- **泛化为"栈无关 + 可插拔规则"的可发布版本**：把核心 checklist 与报告流程从具体技术栈解耦，作为 branch-review-guard 套件的依赖发布（被其综合评审"性能"维度复用）。
- 去具体栈词、改通用表述：`Dubbo` → 远程(RPC)调用；`Redis` → 缓存；`ThreadLocal` + 登录上下文 → "请求上下文（用户/租户/语言/时区）默认不跨线程，提交任务前显式捕获传递"；scheduled / 公共池 → "专用有界线程池而非公共池"；自研事务 / 分布式锁 / 虚拟用户回源等具体机制 → 通用描述（"框架事务边界""分布式锁防击穿""合成/演示数据不触发真实回源"）。
- frontmatter 增加 `version: 0.2.0`；`description` 去栈化、通用化。

### 新增
- 新增「加载栈特有规则」一节：跑通用 checklist 的同时，可加载 `rules/` 中 `dimension: performance` / `correctness` 且 `applies_to` 匹配当前仓库的规则做机制级深度检查；未启用任何栈包时降级为只跑通用 checklist（预期，不报错）。
- 配套通用版 `cursor-rules/endpoint-perf-review.mdc`（Cursor 提醒规则）与 `README.md`。

### 保留
- 栈无关核心不变：证据优先方法（先量化调用清单 → 砍多余 → 批量化 → 缓存 → 并发降级 → 特殊路径 → 测正确 → 留观测）、冷/热缓存最坏情况区分、`EXPLAIN` 等通用 DB 手段、口诀、报告模板（结论先行 / 调用清单 / 发现项 P0/P1/P2 / 推荐方案 / 风险与护栏）、以及"这是提问清单不是必做项，按 ROI 取舍"的取舍原则。

### 说明
- 运行时维度（真实 RT/p99、并发竞态、降级/熔断真实行为）只输出"需运行时验证项"，静态评审不下"已验证通过"结论（与套件 `rules/baseline/runtime-claims-need-evidence` 一致）。
