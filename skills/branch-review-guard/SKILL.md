# Branch Review Guard

本 skill 是一个**评审编排器（orchestrator）**。它不是再造一个"大而全的单体评审脚本"，而是把一次"提测前整分支评审"拆成多个**专项 reviewer**，分批、按风险聚焦地跑完，再汇总成**一份**中文报告，给出可发布性结论。

设计原则（综合 Google eng-practices、SmartBear/Cisco 实证、Meta/微软/亚马逊实践）：

- **设计 > 正确性 > 复杂度 > 测试 > 命名/注释 > 风格**：高价值维度优先，风格交给自动化。
- **大改动不可线性硬读**：>400 行后人/AI 的缺陷检出率断崖下跌；必须分批 + 按风险聚焦 + 显式声明覆盖范围。
- **复用而非重造**：API/兼容/影响走 `api-change-guard`，性能走 `endpoint-perf-review`，本 skill 只补缺失维度并做汇总。
- **栈无关核心 + 可插拔规则**：核心 checklist 不绑定任何框架；技术栈/项目特有的"坑"与"降噪校准"外置到 `rules/` 规则包，运行时加载叠加。
- **诚实边界**：区分事实/推测/待人工确认；运行时维度（性能 p99、并发竞态、迁移在存量数据下表现）只输出"需运行时验证项"，**禁止下"已验证通过"结论**。

## 可移植性（自包含，不依赖任何特定 Agent）

本 skill 是纯"Agent 规则 + Git 命令"，**不依赖任何 IDE 自带的 `/review` 类命令**，可在 Cursor / Claude Code / Codex CLI / 其他能读文件 + 执行 Git 的 Agent 中运行。

- **支持子代理的 Agent**（如 Cursor 的 explore/Task）：每批 / 每维度**并行派一个子代理**，上下文隔离、互不挤占。
- **不支持子代理的 Agent**：顺序多轮，一批做完再做下一批；用同一套 prompt 与 checklist，结果一致，只是更慢。

本套件的发布载体是 Claude Code 插件（`.claude-plugin/`）；非 Claude Code 的 Agent 经安装器（`install/SKILL.md`）落地到 `tools/<name>/` 或直接读 `skills/<name>/SKILL.md`。各形态共享同一份 `skills/` + `rules/` 内核。

## 规则机制（栈无关核心 + 可插拔规则包）

本 skill 的**核心是栈无关的**：所有维度 reviewer 跑的都是通用 checklist（正确性/设计/安全/测试/可观测的一般要点），不绑定任何具体框架。技术栈/项目特有的"坑"与"降噪校准"全部外置到可插拔的 **`rules/` 规则包**，运行时加载、叠加应用。

- **规则包位置**：随 skill 安装在 `rules/`（canonical：`tools/branch-review-guard/rules/`）；启用哪些包由 `rules/config.yaml` 控制。完整 schema 与消费方式见 `rules/README.md`。
- **项目本地规则叠加**（v0.5.0+）：除插件自带 `rules/` 外，评审时 **best-effort 叠加**读取**被评审项目根**的 `branch-review-rules/`（与 `branch-review-reports/` 平行；项目主人自放、通常 `.gitignore` 忽略、纯本地）。这里的规则**独立于 `config.yaml` 的 pack 开关**——不走 `enabled` 判定，直接全量按 `dimension` + `applies_to` 加载，与插件自带规则同等消费（finding 出发现、calibration 降噪）。读不到（目录不存在）就跳过，不影响评审。开发侧 skill 也读这同一份，故**开发与评审标准一致**、单一源不漂移。
- **baseline 包默认开启**（栈无关的通用规则 + 通用降噪校准）；**栈包可选**（如 `rules/skg-spring/`，默认关闭；同栈团队启用后即可获得机制级深度）。团队也可新增 `rules/<your-stack>/` 自定义包。
- **`discover-new/` 是团队沉淀区**（默认关闭）：`distill`/`rule` 反哺闭环产出、人工确认后的规则落在这里，与上游作者预置的 `skg-spring/` **解耦**——升级插件时两者互不覆盖，也便于区分"作者预置 vs 我们实测沉淀"。要生效在 `config.yaml` 手动开。
- **自动识别启用**：`enabled: false` 的栈包若配置了 `auto_enable.project_markers`，加载时按标记探测目标项目（仓库/模块目录名、`pom.xml` 等构建文件的 groupId/artifactId 的确定性字符串匹配），命中则**本次运行自动启用**——不修改 `config.yaml`，报告"已启用规则包"注明"（自动识别启用）"。显式 `enabled: true` 优先；未命中不启用，不做模糊推断。
- **缺包降级**：未启用某栈包时，对应的机制级深度**自然缺席**——这是预期行为；**通用 checklist 照常全跑**，绝不因"没装某包"而报错或中止。
- **reviewer 如何消费规则**（详见 `rules/README.md`）：每个维度 reviewer 在跑通用 checklist 的同时，按 `dimension` + `applies_to`（语言/框架/路径）匹配出本维度已启用规则；对 `type: finding` 规则按"识别要点 + 取证方式"产出发现，对 `type: calibration` 规则按"校准动作"做降噪（直接越过 / 降级）。
- **严重度以规则为准**：最终定级以"规则 `severity` + calibration 动作"为准，可在规则文件或 `rules/config.yaml` 的 `severity_overrides` 覆盖。

## 调用方式

任意能读本文件的 Agent 都可按下面流程执行。建议提示语：

- `读取 tools/branch-review-guard/SKILL.md，对当前分支相对主分支做提测前综合评审`

命令（**Claude Code 插件形态**为命名空间命令 `/branch-review-guard:review`；其它形态如 Cursor/安装器为 `/branch-review-guard`，或直接用自然语言触发本 skill。下列以模式与选项为准）：

- `/branch-review-guard:review` —— 默认 = `branch`，分支相对 base 的累计变更全维度评审（合并前评审，最常用）
- `/branch-review-guard branch [--base <分支>] [--dimensions <维度逗号分隔>]`
- `/branch-review-guard module <模块名>` —— 只深审某个模块（缩小范围、提高深度）
- `/branch-review-guard diff` —— 仅未提交变更（插件形态另有独立命令入口 `/branch-review-guard:diff`，等价 `review diff`）
- `/branch-review-guard recent <N>` —— 最近 N 个提交

`--dimensions` 取值：`bug,design,quality,security,test,semantic,api,perf,observability,i18n`（缺省 = 全部；`semantic` = 业务语义/不变式维度）。

`--thorough`（可选，非默认）：首轮评审后对高风险批次（对外契约/事务并发/鉴权/公共代码）追加"新鲜眼"多轮扫描——**loop-until-dry**：一轮轮派新鲜眼直到**连续 2 轮零新发现**才停（挖干为止，轮数不设人为上限），只保留安全阀 8 轮防"每轮换措辞永不收敛"的死循环（正常撞不到）。轮数**按 diff 自然收敛**：小 diff 常 2-3 轮干涸、大而复杂的 diff 可能跑到 5-6 轮（见 `## 大 Diff 分批，强制全覆盖`）。

配套命令：
- **`/branch-review-guard:design <需求>`**（设计侧姊妹技能）：需求方案设计阶段多视角设计对比，产出对比表 + 推荐方案 + 嫁接综合方案 + 精炼设计稿；设计稿移入 `docs/` 后本评审的「建立上下文」会自动读取。
- **`/branch-review-guard:distill [N]`**：从目标项目本地最近 N 份评审报告聚类重复发现，生成 `rules/` 候选规则草稿（见 `## 反馈闭环（distill）`）。
- **`/branch-review-guard:rule <描述> [--type finding|calibration]`**：把一条已确信的经验**手动**快捷生成一条规则草稿（绕过 distill 的 ≥2 次阈值、由人担保泛化），流程见 `prompts/add-rule.md`。

## 工作流程

> **自主一气呵成，中途不停**：被触发后，**自动连贯执行下面 1→12 全部步骤、直接产出完整报告**，不要在任何中间步骤停下来向用户提问或征求"是否继续/是否开始评审"。唯一允许中止并回问用户的情形：(a) base 分支无法确定（且未给 `--base`）、(b) 命令模式/参数歧义无法解析、(c) 目标范围内**无任何变更**（diff 为空、真的没有代码可评）。除此之外——**包括 diff 很大（数百文件、上万行）**——都按 `## 大 Diff 分批，强制全覆盖` 自动跑完，绝不因体量或"先确认一下"而暂停。下列步骤（含"建立上下文"）都是你**自己执行**的内部动作，不是与用户的交互节点。
>
> **⚠ (c) 只指"diff 为空"，不指"diff 和上次报告一样"**：本次要评审的代码与某份历史报告同一 SHA / diff 字节一致，**不是**中止或跳过的理由——那是正当的重审场景（换模型复审 / 多轮挖掘 / 交叉确认都是"同代码重看"）。**禁止以"无信息增益 / 已评审过 / 代码没变"为由跳过或降级子代理编排**，被触发一次就完整跑一次。详见 `## 每次评审都是独立执行`。

1. **确定范围**：解析命令（默认 `branch`）。按 `## 分析模式` 收集 Git 证据；同时取报告元数据 `git rev-parse --short HEAD` 和时间戳。**顺手做 DLP 明文自检**（取变更清单里一个源码文件用 Read 读，判定是否加密环境，见 `## DLP 透明加密环境对策`），把结论带入后续所有维度的取证方式。
2. **建立上下文（自己读，读完立即继续）**：自行读分支目标 / 需求 / 设计文档（仓库根目录与各模块 `docs/` 下的 `*_DESIGN.md`、`*_CONTRACT.md` 等）、`git log` 的 commit message，在报告里用一段话总结"这条分支在做什么、为什么"。**这是评审的前置自查，不是向用户提问的节点**——读完直接进入下一步，**不要停下来等用户**。（无上下文就裸评是反模式，但"无上下文"靠你自己去读文档解决，而非回问用户。）
3. **自动化先行（L1，见 `## 自动化分级`）**：按项目工具链先尝试编译/构建、lint、测试、依赖漏洞扫描（SCA），把风格与明显问题清掉，让人与 AI 只聚焦逻辑、设计、契约。命令不可用时跳过并在报告"评审范围与方法"中说明。
4. **加载规则包（见 `## 规则机制`）**：读取 `rules/config.yaml`，确定本次启用的规则包（`enabled: true` 的包 + `auto_enable` 标记命中目标项目而运行时启用的栈包）；**再 best-effort 叠加被评审项目根 `branch-review-rules/`**（独立于 pack 开关、全量按 `applies_to` 加载），合并后供各维度 reviewer 按维度匹配应用。
5. **估规模并分批（见 `## 大 Diff 分批，强制全覆盖`）**：`git diff --stat <base>...HEAD` 拿到变更文件总数与全量清单，按模块/业务域切批。
6. **分维度评审**：对每批应用 `prompts/` 下对应维度的 reviewer prompt（见 `## 评审维度与复用映射`），每个 reviewer 同时注入已启用 `rules/` 规则。支持子代理则并行，否则顺序。
7. **复用专项 skill（弹性路径解析）**：API/兼容/影响/回归调用 `api-change-guard`；对识别出的高风险接口调用 `endpoint-perf-review`（解析与降级规则见 `## 评审维度与复用映射`）。
8. **对抗性验证（降误报 + 捞漏报 + 交叉确认，见 `## 对抗性验证与完整性核查`）**：按 `prompts/verify-findings.md` 对所有 P0/P1 发现做 3 视角怀疑者投票（证据核实 / 规则校准 / 触发路径），反驳 ≥2 票否决、恰 1 票降级保留标争议、0 票维持标"已对抗验证"；**并对"正例/做得好/零发现批次"派第 4 视角"假设证伪"怀疑者捞漏报**（翻案须 file:line 反证，从严防误翻）；**每条 P0/P1 另派一个独立 reviewer 交叉确认**（不带原结论从零重看）——≥2 独立命中标 `✓✓交叉确认`、仅 1 命中标 `⚠单源`。P2/Nit 不验。
9. **汇总去重**：按 `prompts/consolidate-report.md` 合并各批/各维度发现 + 对抗验证结果，跨批去重、统一优先级（`P0 阻塞 / P1 必改 / P2 建议 / Nit 可选`），成报告草稿。
10. **完整性核查（防漏，见 `## 对抗性验证与完整性核查`）**：按 `prompts/completeness-critic.md` 对账覆盖率、核查维度完整性与可疑批次；小缺口补审（新 P0/P1 同样过第 8 步），补不动的如实修正覆盖声明；结论写入报告第 13 章。
11. **产出报告**：按 `templates/report-template.md` 在项目内生成单份中文报告——优先 `tools/branch-review-guard/reports/`（安装器路径已建则用之），否则在项目根的 `branch-review-reports/`（不存在即创建，建议加入 `.gitignore`）；命名 `branch-review-guard-<mode>-<shortSha>-<timestamp>.md`。
12. **回复**：先给报告链接，再粘贴完整报告正文，正文后固定附 todo 提炼提示（见 `## 回复约定`）。

## 分析模式

某些 Shell（如 Windows PowerShell）不支持 `&&` 连接，命令需逐行执行。一律先 `git fetch origin`，对比 `origin/<base>`（不用可能过期的本地分支）。

### base 分支探测（branch / recent 模式）

1. 若 `origin/master` 存在，用 `master`；
2. 否则 `origin/main`；
3. 否则 `git symbolic-ref --short refs/remotes/origin/HEAD` 探测；
4. 仍不确定则让用户用 `--base` 指定。

### `branch` 模式（默认）—— 分支相对 base 的累计变更

`git diff` 用三点 `...`（自 merge-base 以来本分支引入的改动），`git log` 用两点 `..`：

```bash
git fetch origin
git diff --stat origin/master...HEAD              # 估规模：变更文件总数与全量清单
git diff origin/master...HEAD                     # 全量 diff（可按项目源码后缀过滤，如 -- '*.java' '*.ts' '*.go'）
git diff --name-only origin/master...HEAD         # 文件清单（用于分批与覆盖率核对）
git log origin/master..HEAD --oneline --no-merges # commit 列表（建立上下文）
```

### `diff` 模式 —— 未提交变更

```bash
git diff                  # 工作区未暂存
git diff --cached         # 已暂存
git diff --name-only
```

### `recent <N>` 模式 —— 最近 N 个提交

```bash
git diff HEAD~<N>...HEAD
git log HEAD~<N>..HEAD --oneline
```

### `module <模块名>` 模式

先按 `branch` 取全量清单，再用 `--name-only` 过滤该模块路径前缀（如 `modules/<service-name>/`），对过滤后的文件全量深审。

> 评审范围**排除测试源码**：按语言惯例忽略测试文件（如 `src/test/**`、`*Test.java`、`*_test.go`、`*.spec.ts`、`__tests__/**`）和仅测试的 diff（测试本身在"测试评估"维度单独看）。

## 大 Diff 分批，强制全覆盖

分支累计 diff 通常很大（数百文件、上万行）。**绝不允许因"文件太多"就只分析一部分、把其余直接列入未覆盖。**

1. **估规模**：`git diff --stat <base>...HEAD` 拿到变更文件总数与完整列表。
2. **分批**：切成多批，每批 ≤ ~20-30 文件。分批切法（按本批特点选最利于推理的一种，不局限于模块切）：
   - **按模块/业务域**（同域入口/数据契约/持久化/业务逻辑一批，利于跨文件推理）；
   - **按 commit**（`git log <base>..HEAD --oneline`，每批一个或几个 commit）；
   - **按调用链**（一个对外入口→service→RPC/DB 的完整链路一批，利于发现 N+1、缺降级、越权链——单按模块切会切碎调用链而漏中段调用）；
   - **单文件逐段深读**：超大文件（如 god class 数千行）不要整文件一次塞给 reviewer（中段注意力衰减、lost-in-the-middle），按方法/逻辑段拆成多段分别深读，再合并。
3. **按风险/类型聚焦深读**（不是均匀用力——但**降权只针对"代码类型"，绝不针对"外围业务模块"**）：
   - **高风险全量深读**：对外契约（HTTP API / RPC / 消息队列 / 对接协议）、事务与原子性、并发与锁、鉴权/越权、删除/类型/枚举/必填字段变更、共享/公共代码改动、配置与 DB 迁移（迁移脚本）。**栈特有的高风险模式**（如特定事务/锁注解、框架装配登记）由启用的 `rules/` 规则包补充识别。
   - **低风险抽样（只限真正低信息量的代码类型）**：getter / 样板 / 纯文案 / 格式 / 生成代码 / 锁文件 —— 抽样 + 依赖自动化。**注意：外围业务模块（不属本功能主链路的 controller/service）不属于"低风险"，不得抽样降权**——它们照样可能有无鉴权端点、全表无界重写、N+1 等真问题（历史教训：运维修数接口、外围健康接口曾漏报）。外围模块的 controller/service 与主链路一视同仁全覆盖。
   - **文件优先级**：入口/接口层 → 数据契约（DTO/VO/Request/Response）→ 跨服务调用（RPC/Client）→ 持久化/转换/业务逻辑 → 其他。
   - **覆盖回执（防中段衰减）**：每批 reviewer 必须回传"已深读文件清单 + 逐文件覆盖说明"，批评家对账。大文件逐段深读的，回传要写明"本文件按 N 段拆读、各段行号范围"，防止整文件只读了开头。
4. **并行/顺序**：支持子代理时每批并行一个子代理；不支持则顺序多轮。
5. **覆盖声明**：报告 `## 分析覆盖范围` 必须写明变更文件总数、批次数、每批覆盖文件数，覆盖率逼近 100%。`## 未覆盖风险` **只**用于真正无法解析的复杂类型/跨仓库调用方，**不得**用来装"因体量没读的文件"。
6. **`--thorough` loop-until-dry 多轮扫描（可选，非默认）**：首轮完成后，仅对高风险批次（对外契约/事务并发/鉴权/公共代码）**一轮轮派"新鲜眼"扫描**——同一套维度 prompt、不带前几轮结论；**连续 2 轮零新发现才停**（挖干为止，轮数不设人为上限）。
   - **为什么连续 2 轮而非 1 轮**：单轮零新发现可能只是这一轮的运气（换个角度又能冒出来），连续 2 轮零新增才说明真挖干了。这比"跑满固定轮数"更聪明——该停时早停、该多跑时放开跑，让 diff 复杂度决定轮数（小 diff 常 2-3 轮、大而复杂的可能 5-6 轮）。
   - **安全阀**：只保留 8 轮硬上限，纯防"模型每轮换措辞导致永不收敛"的死循环（正常永远撞不到）；达阀即停并在报告注明"达安全阀、可能仍有长尾"。这是防呆、不是成本约束。
   - **为什么加 loop-until-dry**：召回的长尾——有些发现要换角度扫第 3、4 遍才冒出来（同事 114-agent 的召回优势主要来自 pass 累积）。
   - **天花板（诚实边界）**：同模型的相关性盲点**不随轮数消失**——对"注意力没聚焦"型漏报（如大文件中段 N+1）有效，对"不懂业务道理"型漏报（业务语义类）收益很快见顶。想突破这层要靠换模型（见 `## 使用建议：异构复审（强独立召回）`）。
   - **去重对照"所有见过的发现"（含被对抗验证否决的），不是"已确认的"**——否则被否发现每轮复活、永不收敛。**收敛判据 = 连续 2 轮相对"所有见过的发现"并集零新增**（新增须是经对抗验证未被当场否决的，防止靠复活被否项假装还在产出）。

## 对抗性验证与完整性核查（质量编排层）

分批全覆盖解决"每个文件都读过"（广度），本层解决另两类失败：**看似合理但错**（误报）与**读过但没看出来 / 评审自身漏项**（漏报）。

- **对抗性验证**（`prompts/verify-findings.md`，插件形态派 `bru-skeptic` 子代理）：分两个方向——**(a) 降误报**：每条 P0/P1 派 3 个**不同视角**的怀疑者——①证据核实（file:line 真实且语义如所述？）②规则校准（是否本应被 calibration 豁免/降级？）③触发路径（影响在真实调用路径上成立？severity 高估？）。**有效反驳必须给 file:line 级反证或规则 id**，"感觉不严重"按维持计。聚合：反驳 ≥2 票否决（移入报告第 14 章附录留证，供 distill 沉淀校准规则）、恰 1 票降级一档保留标 `⚠有争议`、0 票维持标 `✓已对抗验证`。**(b) 捞漏报（反向证伪正例）**：对 reviewer 标"做得好/正例/已复核通过"的项、以及高风险批次的"零发现"声明，派第 4 视角**假设证伪**怀疑者主动找反证（典型如"幂等靠 DuplicateKey 但全仓无唯一索引"、"单设备不变式某条路径绕过"）；翻案须 file:line 级反证，从严防误翻真·正例，absence 类反证不用 grep（DLP 假零）。阻塞清单只收经验证的发现，基本免人工甄别。P2/Nit 不验（不阻塞发布，成本不划算）。
- **P0/P1 交叉确认**（借鉴同事 114-agent 的"双人 2/2"，覆盖**所有 P0 与 P1**，与上面的怀疑者投票**正交叠加**——投票查"这条发现对不对"，本层查"这条发现有几个独立 reviewer 都看到了"）：每条 P0/P1 用**另一个独立 reviewer**（新鲜眼、用对应维度 prompt、**不告诉它原发现结论**）从零重看该处代码，看是否独立命中同一问题。它衡量的是**独立命中数**，不是对错——所以与投票正交、要叠加而非二选一。
  - **≥2 个独立 reviewer 命中**（原 reviewer + 交叉确认 reviewer）→ 标 `✓✓交叉确认`（比单源更硬，阻塞清单可直接采信）。
  - **仅 1 个命中**（另一个独立 reviewer 未复现）→ 标 `⚠单源` 保留在阻塞清单，但提示"仅单一 reviewer 命中、建议人工复核"（可能是单 reviewer 的过度解读）。
  - **双向增益**：不只提精度——为"确认"而派的独立 reviewer 重看那片代码时，常**附带捞回新发现**（新发现照走对抗验证）。
  - **范围**：P0 与 P1 都做（漏报/误判在 P1 同样常见——如同事 R2–R6 均为 P1）；P2/Nit 不做（不阻塞发布、边际收益低），仍只走怀疑者投票。诚实边界：同模型的两个 reviewer 独立性有限（`model: inherit`），"交叉确认"是弱独立而非真异构——真正的强独立要靠**换模型多轮**或**人工复核**（见 `## 使用建议：异构复审（强独立召回）`）。
- **完整性核查**（`prompts/completeness-critic.md`，插件形态派 `bru-critic` 子代理）：报告定稿前独立核查"这次评审本身"——覆盖率对账（声明 vs 批次回执并集）、维度完整性（有无静默跳过）、可疑的零（高风险批次零发现不默认干净）、待确认遗漏、诚实边界表述。小缺口补审、补不动的如实修正声明，结论写入第 13 章。
- **非子代理环境**：以上都退化为编排器按同一 prompt 顺序自查（P0/P1 交叉确认退化为"换一套视角重看一遍该发现"），结果一致只是更慢、独立性更弱。

- **业务语义 / 不变式** —— `prompts/review-business-invariant.md`：专捞"代码特征抓不到"的漏报——哨兵值被通用逻辑误判、功能孪生防护不对称（差分）、不变式多路径绕过、删除/清理留孤儿、响应可区分性/枚举 oracle。靠**主动提问 + 差分对照**发现（先列对象清单再套五模式），不靠模式匹配；产出"正例·待证伪"声明对接对抗层第 4 视角。这类问题（如 R7/R8/R10/R12）是静态分析的结构盲区。

## 自动化分级（护栏：禁止越权下结论）

- **L1 可自动定论**：风格/格式/命名、明显空指针/资源未关闭、API/兼容结构化 diff、依赖 CVE、密钥硬编码、SQL 拼接注入模式、测试是否存在/覆盖率数字、i18n 缺失 key。→ 直接给结论。
- **L2 AI 半自动**：设计/架构合理性、复杂度/重复、事务/缓存/锁/越权等**模式化坑**、N+1/调用链放大。→ 给"证据(file:line)+优先级"发现，并标"**待人工确认**"。
- **L3 必须运行时证据**：真实性能（QPS/RT/p99、火焰图、执行计划/`EXPLAIN`、压测）、并发竞态、降级/熔断真实行为、迁移在存量数据下表现与回滚可行性。→ **只输出"需运行时验证项"，严禁给出"已验证通过"。**

## 校准规则（降噪，由 rules/ 提供）

误报校准**不再内联在本文件**，而是由 `rules/` 中 `type: calibration` 的规则提供（baseline 通用降噪默认开；技术栈/团队惯例类校准放在对应栈包，如 `rules/skg-spring/`）。

- reviewer 与汇总阶段按命中的 calibration 规则的"校准动作"处理：**直接越过 / 降级**，是否计入 P0/P1、是否进"待人工确认"均以规则为准。
- 校准只为**降噪**，**不放松**对真实缺陷与对外/C 端接口越权的判定。
- 想增删/调一条校准：编辑 `rules/<pack>/` 下对应规则的 `enabled` / `severity`，或在 `rules/config.yaml` 覆盖——无需改本文件或 prompt 主体。

## DLP 透明加密环境对策（横切纪律）

> 这是**横切纪律**，影响每一个维度、每一个子代理。某些主机装了 DLP（数据防泄漏）透明加密，工作区文件**内容**被加密（文件名不加密）。于是任何按文件**内容**搜索的工具——Bash 的 `grep`/`rg`/`sed`、Grep 工具、外部静态分析引擎——读到的都是密文，返回**"假零命中"**。它双向都有害：reviewer 找调用方会"搜不到"→漏报；怀疑者核实证据会"找不到反证"→把真发现误杀。这次的插件报告靠 Read 工具走解密层兜住了，但那是约定、不是机制。本节把它变成机制。

1. **开局明文自检（工作流程第 1 步"确定范围"里顺手做）**：从变更清单取一个已知源码文件（`.java`/`.ts`/`.go` 等），用 **Read** 读出来，检查内容是否以 `%TSD-Header` 开头、或含大量 NUL 字节。
   - 是 → 本环境 **DLP 加密**，在报告"评审范围与方法"声明「DLP 透明加密环境：内容检索已切 Read-only 模式」。
   - 否 → 正常环境，Grep 可正常用于内容检索。
   - Read 出来仍是密文（连 Read 也拿不到明文）→ 报告顶部红字警告「**本环境致盲、召回不可信，建议换非加密环境重跑**」——诚实边界优先，不假装查过。
2. **内容检索去 grep 化（DLP 加密时）**：**文件名用 Glob、文件内容用 Read**；**禁止用 Bash 的 `grep`/`rg`/`sed` 或 Grep 工具按内容检索**。已提交文件可用 `git show HEAD:<path>` / `git cat-file --filters <base>:<path>` 读 git 对象库里的明文做基线交叉核对（git 对象库不受工作区加密影响）。
3. **否定型结论必须标注取证方式**：所有否定型断言（"无其它调用点 / 不存在 / 全仓无 / 未发现 X"）必须附取证标记 `[Read]` / `[git]` / `[Grep·仅非DLP可信]`。DLP 下凡 `[Grep]` 取证的否定结论**一律不可信**，由完整性核查（`completeness-critic.md`）强制退回 Read 重核；拿不出 Read 复核的，改写为「未核实」而非「不存在」。
4. **子代理同样适用**：派发给 `bru-*` 子代理的任务书里注入本节纪律；怀疑者（`bru-skeptic`）的"证据核实"视角、批评家（`bru-critic`）的取证核查，都以本节为准——子代理白名单里的 Grep 工具在 DLP 下只用于"已知非加密"或路径/文件名类查找，不用于内容否定取证。

## 每次评审都是独立执行（禁止"同 SHA 无增益"投机）

> 这是**横切纪律**，防一类隐蔽的自作主张：agent 发现"本次要评审的代码与某份历史报告是同一 commit / diff 字节一致"，就臆断"代码没变、重跑无信息增益"，于是**跳过或降级子代理编排**——改成只 Read 复核历史结论、只看增量 commit。**这是错的，禁止。**

**为什么禁止**：本套件的召回恰恰建立在"**同一份代码、换个推理者或换个角度再看一遍，会看出上一遍没看出的东西**"这个前提上——这正是同事 114-agent 胜过 12-agent 的原因。三个核心机制全依赖"真的重新跑一遍"：
- **异构复审**（`## 使用建议：异构复审`）：换模型后重跑，SHA 没变但**推理者变了**，正是要拿独立第二意见——若因同 SHA 跳过，等于复用旧模型结论，换模型形同虚设。
- **loop-until-dry**（`## 大 Diff 分批` 第 6 步）：同 SHA 多轮新鲜眼，靠的就是"同代码多看几遍"。
- **P0/P1 交叉确认**（`## 对抗性验证与完整性核查`）：另派 reviewer 从零重看同一处代码。

**纪律**：
1. **同一 SHA / diff 与历史报告一致，是正当的重审场景**（换模型复审、多轮挖掘、交叉确认都是"同代码重看"），**绝不**以"无信息增益 / 已评审过 / diff 未变 / 代码没变"为由**跳过或降级** `## 工作流程` 的任何步骤或子代理编排。被触发一次，就完整跑一次编排。
2. **历史报告只能当参考上下文，不能替代本次真实执行**：可以读上一份报告帮助建立上下文（第 2 步），但维度评审、对抗验证、完整性核查必须**本次真跑**，不允许"复用上轮结论"充当本次产出。
3. **唯一能因"无变更"中止的**，仍只有 `## 工作流程` 中止条件 (c)——**目标范围内 diff 为空**（真的没有任何代码可评）。"diff 和上次报告一样"**不等于**"diff 为空"：前者有代码要评、只是被评过，仍须重跑；后者才是空、才中止。别把两者混为一谈。

## 使用建议：异构复审（强独立召回）

> 这是**使用实践、不是插件内的自动流程**——因为插件子代理是 `model: inherit`，一次评审内部无法混用不同底模。想要真正的"独立第二意见"（对标同事 114-agent + 双人 2/2 的强独立），要靠**主会话换模型跑多轮 + 人工复核**。以下是 SOP，团队照做即可。完整可传阅版见 `docs/异构复审-SOP.md`。

**为什么需要**：插件内的 loop-until-dry 和 P0/P1 交叉确认都是**同底模**的改良，天花板是"同模型相关性盲点"——同一个模型盯不到的语义坑，跑多少轮、派多少个同款 reviewer 都还是盯不到。突破这层的唯一办法是引入**真正不同的推理者**：另一个架构/厂家的模型，或人。

**SOP（异构复审四步）**：
1. **主审**：用当前模型正常跑 `/branch-review-guard:review`，得报告 A。
2. **换模型**：`/model` 切到**架构或厂家不同**的模型（同家族盲点太接近、换了等于没换）。可切项以 `/model` 实际列出为准。
3. **补审**：换模型后重跑，最省成本的是只补最吃多样性的维度 `--dimensions semantic,security,bug`；不计成本则整份重跑得报告 B。
   - **⚠ 防投机**：换模型后 SHA 没变，若 agent 说"同一 SHA、diff 一致、无需重跑"——**那是投机行为，不接受**。异构复审的价值就在"同代码换个模型再看"，明确要求它按换后的模型**真实重跑**（依据 `## 每次评审都是独立执行`）。
4. **对照合并**：两份都报的同一问题 = **高置信**；只有 A 或只有 B 报的 = **异构捞回的候选**，标"单源、待人工复核"。
5. **人工兜底（最强异构）**：对 Top P0/P1 自己或拉同事独立看一遍——**人是当前能用的最强"不同模型"**，这就是同事"双人 2/2"的精髓。

**诚实边界**：换模型能捞回"注意力/推理路径不同"型漏报（第一层），但捞不回"根本不知道这条业务道理"型漏报（第二层，如墓碑哨兵/单设备不变式是不是产品要求）——后者必须靠人喂领域知识（领域不变式台账，见 `prompts/review-business-invariant.md`）或需求/设计文档注入。

## 可靠性规则

- Git diff 与文件内容是事实依据；区分**事实 / 推测 / 待人工确认**。
- 不编造调用方；影响范围给依据与概率。
- 每条发现可定位（`file:line`）+ 可执行建议；区分阻塞与 Nit。
- 显式声明覆盖率与未覆盖，**绝不把没读的当已评审**。
- 同时肯定"做得好的地方"（Google: Good things），基调是"持续改进而非追求完美"。
- 注释掉的接口/路由代码不视为真实接口。
- 运行时维度只给"需运行时验证项"，不下"已通过"。

## 反馈闭环（distill + 手动 rule）

评审的历史产出可反哺规则包与开发侧，两条入口共享同一套落地关卡（草稿落报告目录旁 `rule-drafts/`、`enabled: false`、**人工确认后**提交插件仓库 `rules/<pack>/` 才生效，**绝不自动写规则包**）：

- **`/branch-review-guard:distill [N]`**（数据驱动，流程见 `prompts/distill-rules.md`）：读**目标项目本地**最近 N 份报告聚类重复发现。**先把每条发现归到"代码实例"（同 file + 同根因，不死磕 file:line）再计数，同一实例跨报告只计 1 次**——漏报模式（同维度同根因跨 **≥2 个不同实例**）→ 候选 `finding`；对抗验证附录中反复被否决的模式 → 候选 `calibration`。
  - **遗留项分诊**：同一实例在 ≥2 份报告中反复报出、位置基本未变的，**不是漏报**（评审每次都报了、是开发一直没改），从 finding 剔除、单列为「反复报出但未修复的遗留项」，给两个出口——认可是真问题只是没排期 → known-issue 不生成规则；判定不成立/不重要 → 转 `calibration` 让评审器以后豁免。**避免把"一直没改的老问题"误固化成 finding**。
- **`/branch-review-guard:rule <描述> [--type ...]`**（人担保，流程见 `prompts/add-rule.md`）：把一条已确信的经验**手动**快捷生成规则草稿，**绕过 distill 的 ≥2 次阈值**（泛化由录入人负责）。典型用途：把 distill 遗留项一句话转 calibration，或一眼确信要规则化的强 case。

开发侧 skill / CLAUDE.md 引用同一套 `rules/` 的"修法"节，即可把评审教训前置为写码禁区。
