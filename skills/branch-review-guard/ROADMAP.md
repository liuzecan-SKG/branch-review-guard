# Branch Review Guard — 迭代计划

> 来源：基于对真实大分支报告（约 250 文件量级）的产出质量分析，以及与 IDE 自带的 `/review` 类命令和市面工具（CodeRabbit / Greptile / Qodo·PR-Agent / Copilot code review / Graphite Diamond / SonarQube / CodeGuru）的三方对比。
>
> 已立住的差异化：**整分支累计 + 强制全覆盖 + 栈特有坑机制级识别（经可插拔规则）+ L1/L2/L3 诚实护栏 + 可发布性裁决 + 自包含可移植 + 中文提测报告**。后续目标是**补齐对比中明显的短板**（增量/速度、PR 原生集成、确定性扫描、低误报、可审计覆盖），同时**不牺牲**上述差异化与可移植性。

## 设计原则（必须守住）

- 保持**自包含、零运行时依赖、可移植**：新增能力一律可选、可降级；缺环境时回退到基础行为并明确声明。
- 保持**栈无关核心 + 可插拔规则**：技术栈特有知识只进 `rules/` 规则包，不回流到核心 prompt。
- 保持**诚实边界**：L3 运行时维度永不下"已验证通过"。
- 保持**正本单一**：`tools/branch-review-guard/` 为正本，改完同步 `.cursor` / `.claude` 镜像。
- 默认 `branch` 全量模式**行为不变**，作为提测/上线卡点；新模式/集成均为叠加项。

## 里程碑总览

- **已交付（0.2.0）**：规则外置化（#3）、弹性路径解析、栈无关化。
- **M1（低成本高收益，先做）**：覆盖率可审计（#6）、增量模式（#1）。
- **M2（工程化集成）**：确定性 SAST/SCA 串联（#5）、CI 化 + PR 评论/门禁 + SARIF（#2）、误报反馈闭环（#4）。

优先级标注：`[P0]` 必做、`[P1]` 应做、`[P2]` 视资源做。

## 已交付（0.2.0）

### #3 规则可配置化 + 严重度校准 ✅

- **痛点**：栈特有坑与误报校准曾硬编码在 prompts，团队无法按信噪比逐条增删/调级。
- **已做**：新增 `rules/` 目录，把"栈特有坑 + 误报校准"抽成**版本化、可逐条开关、可调默认严重度、可按栈匹配**的规则条目（frontmatter：`id / pack / type / dimension / severity / enabled / applies_to / summary` + 正文方法）。`baseline` 默认开、栈包（如 `skg-spring`）可选。SKILL/prompts 改为"读取 `rules/` 应用"，而非内联硬编码。
- **弹性路径解析**：复用 `api-change-guard` / `endpoint-perf-review` 时按 `tools/` → `.cursor/skills/` → `.claude/skills/` 解析，缺失则降级声明，修复了旧版硬编码路径的 bug。

## M1

### #6 覆盖率机械可验证 `[P0]`

- **现状痛点**：报告里"N/N 深读、约 100%"是模型自报，不可外部审计，存在"声称读了但推理不到位"的盲区。
- **做什么**：每个维度子代理/批次输出**实际读过的文件清单**（结构化）；汇总阶段用 `git diff --name-only origin/<base>...HEAD` 做差集，自动核对"已读 ∪ 未读 = 全量 diff"，算出真实覆盖率并列未读文件。
- **落地方式**：新增 `prompts/verify-coverage.md`；改 `prompts/orchestrate-branch-review.md`（要求每批回传文件清单）、`prompts/consolidate-report.md`（差集核对）、`templates/report-template.md` 第 13 章（附核对结果）。
- **验收**：报告第 13 章给出机器核对的覆盖率与未读文件清单，且 `已读 + 未读 == diff 全量`；若不等则报告显式告警。

### #1 增量 / 按 commit 模式 `[P1]`

- **现状痛点**：每次都是整分支重跑、无增量；迭代期（边改边评）token 与耗时成本高。
- **做什么**：新增 `incremental` 模式，缓存上次评审的 `commit SHA + 覆盖清单`，只评"自上次以来新增的 commit/文件"；引入类 patch-id 去重。保留 `branch` 全量模式作为提测卡点。
- **落地方式**：改 `SKILL.md` 增 `/branch-review-guard incremental [--since <ref>]` 与状态文件约定（如 `reports/.state.json`，纳入 `.gitignore`）；`consolidate` 合并历史结论与本次增量。
- **验收**：第二次跑只处理增量文件，耗时/token 显著下降；`branch` 全量行为不变。
- **风险/取舍**：增量会牺牲"全覆盖契约"——文档明确"增量用于迭代期、提测前必须跑一次全量"。

## M2

### #5 确定性 SAST / SCA 串联（避免 L1 降级）`[P1]`

- **现状痛点**：L1（编译/测试/依赖扫描）依赖本机环境，环境不全时退化为纯静态评审，CVE/密钥/注入这类**应确定性穷举**的维度被弱化。
- **做什么**：把 L1 固定到容器化确定性工具链——Semgrep（SAST）、Trivy / OWASP dependency-check（SCA/CVE）、gitleaks（密钥），可选 SonarScanner；其结构化结果作为 L1 输入并入报告，把 LLM 算力释放到事务/并发/设计等它真正擅长的推理上。
- **落地方式**：新增 `prompts/run-static-gates.md` + `scripts/`（容器化运行命令与结果归一化说明）；改 `SKILL.md` L1 段（优先用确定性工具，缺失再降级并声明）。
- **验收**：一键产出 Semgrep/Trivy/gitleaks 结果并并入报告 L1；环境缺失时明确降级声明。
- **风险/取舍**：引入外部工具会削弱"零依赖"——设为**可选增强**，缺失时回退纯静态评审，不破坏可移植性。

### #2 CI 化 + PR 评论 / 门禁 + SARIF `[P1]`

- **现状痛点**：产物是一份长报告，无 PR 行内评论、无 status check、无合并门禁——发现靠人读、人分诊、人跟进。
- **做什么**：提供 CI 入口（GitHub Actions / 通用），在 PR/MR 上跑，把发现以**行内评论 + 一个 status check**落地（经 `gh api`），P0/P1 未解决则 `fail`；可选输出 **SARIF** 接 code scanning。中文长报告仍作为存档产物。
- **落地方式**：新增 `ci/`（workflow 模板 + 脚本说明）+ `README` CI 章节；`consolidate` 增加 SARIF/评论两种输出适配。
- **验收**：PR 上能看到 status check + 行内评论 + 存档报告；P0/P1 能卡 merge。
- **风险/取舍**：**前置依赖** CI 中可跑 AI Agent（Cursor CLI / Claude Code CI / Codex）。无 Agent 运行时则只产出报告工件，不强求门禁。

### #4 误报反馈闭环 `[P2]`

- **现状痛点**：只有手工"误判记录"，无法系统性降噪；反复被忽略的模式仍重复报。
- **做什么**：把"误判记录"升级为**结构化反馈**——开发对每条发现标 `采纳 / 忽略 / 误报`；累积后对高频被忽略的模式产出"降级/关闭"建议，回写到 `rules/`（半自动，人确认后生效）。
- **落地方式**：新增 `feedback/` 模板 + 周期性汇总 prompt；`consolidate` 读取历史反馈做即时降噪；处置记录留在 `rules/<pack>/FEEDBACK.md`。
- **验收**：能记录处置并对高频忽略模式产出降噪建议；建议可一键并入 `rules/`。
- **依赖**：建立在 #3（规则外置，已交付）之上。

## 不做 / 暂缓（明确边界）

- **全托管 SaaS、看板/分析、SSO/审计、多仓库策略**：非本地自包含工具定位，不做。
- **跨 PR 的机器学习式自学习**：只做"规则化 + 人确认"的轻量闭环（#4），不做模型级学习。
- **IDE 内实时评审**：实时性交给 IDE 级工具，本工具仍定位"提测/上线前一次性综合评审"。
- **跨仓库 RPC 消费方索引**：仍标"待人工确认"，不自建全仓索引。

## 成功度量

- **可审计覆盖率**：报告覆盖率由"自报"变"机器核对"，未读文件可列。
- **迭代成本**：`incremental` 模式相对全量，迭代期耗时/token 明显下降（目标 ≥50%）。
- **误报率**：规则外置 + 反馈闭环后，被开发标"忽略/误报"的发现占比下降。
- **可门禁**：能在 PR 上以 status check 卡住未解决的 P0/P1。
- **不回退差异化**：全量 `branch` 模式的全覆盖契约、机制级识别、L3 诚实护栏保持不变。

## 兼容性

后续能力全部为**叠加式增量**：默认 `branch` 全量模式行为不变；`incremental` / CI / 静态闸门 / 反馈均为可选项，缺失时回退基础行为。规则包按需开关，缺包只跑通用 checklist。正本仍在 `tools/branch-review-guard/`，`SKILL.md` 改动同步两个镜像。
