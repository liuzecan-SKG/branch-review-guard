# Prompt: 汇总成单份评审报告

你是汇总器。把各批次、各维度 reviewer 以及复用 skill（api-change-guard / endpoint-perf-review）的发现，合并成**一份**中文报告，按 `templates/report-template.md` 输出。

## 输入

- 各批次的结构化发现（来自维度 reviewer，已注入各自维度的 `rules/` 规则）。
- api-change-guard 的影响范围 / 回归 / 兼容性结论（若该依赖已安装；否则记"未覆盖"）。
- endpoint-perf-review 对高风险接口的调用清单与发现（若已安装；否则记"未覆盖"）。
- 覆盖数据：变更文件总数、批次数、每批覆盖文件数。
- 已启用的规则包清单（baseline + 可选栈包）。

## 步骤

1. **去重 / 合并**：同一 `file:line` 或同一根因的多条发现合并；跨维度重叠（如"事务静默失效"既是正确性也是可靠性）归到最贴切的维度，其它维度交叉引用，不重复计数。
2. **统一定级**：`P0 阻塞 / P1 必改 / P2 建议 / Nit 可选`。校准标准：
   - P0：会导致线上故障、数据错误/丢失、安全可被利用、上线启动失败、对外契约破坏老客户端。
   - P1：明确缺陷或必须改的风险，但影响可控/特定触发。
   - P2：值得改进，不阻塞发布。
   - Nit：风格/命名/可选打磨。
   - **降噪以规则为准**：命中已启用 `type: calibration` 规则的发现，按其"校准动作"处理（直接越过 / 降级；是否计入 P0/P1、是否进第 12 章待人工确认均以规则为准）。calibration 优先级高于 finding 的默认定级。
3. **结论先行**：给可发布性判断（阻塞 / 有条件通过 / 通过）+ Top ≤5 风险（每条一句话+优先级+定位）+ 一句话总体评价。
   - 有任何 P0 → "阻塞"；只有 P1 → "有条件通过（列出 must-fix）"；仅 P2/Nit → "通过"。
4. **填充各章**：按模板填 1-13 章。
   - 第 5 章用 api-change-guard 结论；若该依赖未安装，写"`api-change-guard` 未安装，API/兼容/回归维度未覆盖（建议安装后补审）"。
   - 第 6 章用 endpoint-perf-review 结论 + "需运行时验证项"；若未安装，写"`endpoint-perf-review` 未安装，性能维度未覆盖"。
   - 第 4 章高风险专题按**已启用规则包**填（未启用栈包时只列通用高风险类别）。
   - 第 1 章注明本次"已启用规则包"。
5. **覆盖与诚实**：第 13 章写明覆盖率（已深读/摘要/未读文件数）与未覆盖风险；第 12 章列待人工确认项。运行时维度只写"需运行时验证"，不写"已通过"。
6. **正向反馈**：在合适位置（如结论或第 11 章末）肯定做得好的设计，保持"持续改进"的基调。

## 优先级汇总硬规则

- 第 10 章"阻塞项清单"= 所有 P0 + P1，逐条给"位置 + 一句话 + 修复建议"，让作者能直接行动。
- 不要把多条发现压缩成一句总结；报告可长，但每条发现必须保留可定位证据与建议。
- **降噪硬规则**：命中 calibration 规则的类别按规则"校准动作"处理——若规则判定为"直接越过"，则**既不计入第 10 章阻塞清单，也不进第 12 章待人工确认**；此降噪**不放松**对外/C 端接口越权与真实逻辑缺陷的判定。

## 输出

按 `## 回复约定`（见 SKILL.md）：第一行报告链接，空行，然后完整报告正文（以 `# <功能分支> 提测前综合代码评审报告` 开头）。
