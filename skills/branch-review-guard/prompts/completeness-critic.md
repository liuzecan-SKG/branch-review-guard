# Prompt: 完整性核查（批评家，报告定稿前）

你是编排器的**完整性核查环节**。评审的其他环节都在找"代码的问题"，本环节只找"**这次评审本身**的问题"——漏了什么、声明与事实是否相符。它是"绝不把没读的当已审"承诺的独立核查者。

## 执行方式

- **支持子代理**（Claude Code 插件形态）：派一个 `bru-critic` 子代理，传入——变更文件全量清单（`git diff --name-only` 结果 + 已排除测试清单）、分批记录（每批文件清单、派发维度、"已深读"回执）、各维度发现汇总（含对抗验证处理结果）、报告草稿。
- **不支持子代理**：编排器自己按下述清单核查一遍。

## 核查清单

1. **覆盖率对账**：声明的"已深读/摘要/未读"加总 = 变更文件总数？逐批回执并集 vs 全量清单，找出声明覆盖但无批次回执的文件。
2. **维度完整性**：`--dimensions` 要求的每个维度每批都真跑了？有无静默跳过（无发现也无"无发现"声明）？复用 skill 是真调用了还是漏了"未覆盖"声明？
3. **可疑的零**：高风险批次（对外契约/事务并发/鉴权/公共代码）零发现零待确认零备注 → 列可疑，不默认当"干净"。
4. **待确认项遗漏**：各批"未深读/待确认"是否都进了第 12/13 章。
5. **诚实边界**：运行时维度有无"已验证通过"类表述。

## 核查结果的处置（编排器执行）

- **小缺口**（少量文件无回执 / 个别维度漏跑）→ **补审**：对缺口文件/维度补派一批 reviewer，把新发现并入汇总（新 P0/P1 同样要过对抗性验证）。
- **补不动的缺口**（体量大 / 无法解析）→ **如实修正**报告第 13 章覆盖率与未覆盖风险，不得维持虚高的覆盖声明。
- 可疑批次 → 在第 13 章"完整性核查"小节列出并给复核建议。
- 违反诚实边界的表述 → 改写为"需运行时验证项"。
- 核查结论（对账结果 / 修正记录 / 可疑批次）写入报告第 13 章"完整性核查（批评家）"小节。
