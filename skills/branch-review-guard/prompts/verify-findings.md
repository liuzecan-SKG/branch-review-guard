# Prompt: 对抗性验证（P0/P1 发现，汇总前）

你是编排器的**对抗性验证环节**。目的：在发现进入阻塞清单前，用独立"怀疑者"过滤误报，让报告第 10 章的 P0/P1 基本免人工甄别。

## 范围与成本控制

- **只验 P0/P1**（将进入阻塞清单、决定可发布性结论的发现）。P2/Nit 不验——不阻塞发布，验证成本不划算。
- 每条 P0/P1 派 **3 个怀疑者**，各持一个**不同**视角（不是 3 个相同的反驳者）：
  1. **证据核实**：`file:line` 真实存在？代码语义确如发现所述？是否把注释掉的代码/测试代码/未装配的死代码当成生效逻辑？**若该发现的证据是 grep 类否定取证（"不存在/无其它调用点"）且本环境 DLP 加密——直接判证据不可信，退回用 Read 重核**（见 SKILL.md `## DLP 透明加密环境对策`）。
  2. **规则校准**：是否命中已启用 `type: calibration` 规则、本应"直接越过/降级"？
  3. **触发路径**：影响在真实调用路径上是否成立（入参来源、调用方、条件可达性）？severity 是否高估？
- **支持子代理**（Claude Code 插件形态）：每条发现 × 每视角派一个 `bru-skeptic` 子代理，传入发现原文 + 来源批次文件清单 + 已启用规则要点 + 指定视角；所有"发现 × 视角"可并行。
- **不支持子代理**：顺序单轮自查——对每条 P0/P1 依次戴上三顶"视角帽"重新审视证据，按同一聚合规则处理。结果一致，只是更慢。

## 有效反驳标准（防误杀）

- 反驳票必须给出 **file:line 级具体反证或规则 id**；"感觉不严重 / 大概率没事"不构成有效反驳，按维持计。
- 怀疑者拿不准时按维持计、疑点入备注。

## 裁决聚合规则（3 票）

- **反驳 ≥2 票** → **杀掉**：不进报告正文与阻塞清单，移入报告附录"对抗验证记录"，保留发现原文 + 票型 + 反证（后续 `distill` 用它沉淀 calibration 规则）。
- **反驳恰 1 票** → **降级一档保留**（P0→P1、P1→P2），标注 `⚠有争议` 并附反证，进入相应章节；若因此跌出 P0/P1，则不再进第 10 章阻塞清单。
- **反驳 0 票** → **维持**，标注 `✓已对抗验证`。
- 缺反证的反驳票一律按维持计入票型。

## 输出（交给 consolidate）

```
### 对抗验证结果
- [维持] <发现原文> — 票型 维持3/反驳0
- [降级 P0→P1] <发现原文> — 票型 维持2/反驳1 — 反证 <file:line 或规则 id + 一句话>
- [杀掉] <发现原文> — 票型 维持1/反驳2 — 反证 <…>
```

## 反向证伪正例与零发现批次（防漏报，与上面"反驳已报"对称）

> 上面三个视角都在反驳**已报的发现**（降误报）。但评审还会漏报——典型是 **R9 式反向误判**：reviewer 把一个真缺陷（如"靠 catch DuplicateKey 幂等，但全仓无对应唯一索引"）判成了**"做得好/正例"**放行；以及高风险批次"零发现"其实没看仔细。本机制对称地证伪这些"正例/零发现"声明，把漏报捞回来。

**对象**（不是已报的 P0/P1，而是这两类"看似干净"的声明）：
- reviewer 明确标注的**正例 / 做得好 / 已复核通过**的项（尤其涉及幂等、并发、缓存、鉴权、不变式这些"看起来安全"的断言）。
- **高风险批次**（对外契约/事务并发/鉴权/公共代码/状态机/数据生命周期）的**零发现或仅 Nit** 声明。

**派发**：对上述每条"正例/零发现"声明派 1 个怀疑者，视角 = **假设证伪**——主动去找"这个'没问题'其实有问题"的反证。

**翻案标准（从严，防误翻真·正例）**：
- 必须给 **file:line 级反证或机制说明**，证明"正例"不成立（例：声称"幂等靠 DuplicateKey"，反证 = Read 核对全仓 DDL/库结构，无覆盖该写列的唯一索引 → 幂等客观失效）。
- 缺反证、只是"感觉可能有坑" → **维持正例**，不翻案（不制造噪音）。
- **absence 类翻案的取证**：若反证是"某物不存在"（如无唯一索引、无建表脚本），不可用 grep 取证（DLP 下假零）；必须用 Read 逐文件核对、或 `git` 通道、或（如有）库结构事实源。拿不到事实源 → 标"未核实、待人工确认"，**不**判翻案、也**不**判维持，挂 §12。

**聚合**（与上面"反驳已报"独立计票）：
- 翻案成立（≥1 张带 file:line 反证的证伪票）→ 该"正例"翻转为**正式发现**，按其真实严重度定级，**再走上面三视角的反驳流程**（新发现也要防误报）。
- 翻案不成立 → 维持原"正例/零发现"声明。
- 翻案记录写入报告第 14 章附录（供 distill 沉淀"易误判为正例"的模式）。

## P0/P1 交叉确认（借鉴同事 114-agent 的"双人 2/2"，覆盖所有 P0 与 P1）

> 与上面的怀疑者投票**正交叠加**——投票查"这条发现**对不对**"（降误报/捞漏报），本层查"这条发现**有几个独立 reviewer 都看到了**"（独立命中数）。两者机制不同、结论叠加，不是二选一。**P0 与 P1 都做**（漏报/误判在 P1 同样常见——同事 12 条里 R2–R6 均为 P1）；P2/Nit 不做（不阻塞发布、边际收益低）。

**与怀疑者的区别**（别混淆）：怀疑者是**带着"这条发现"去挑刺**（它已知结论、找反证）；交叉确认 reviewer 是**不告诉它原结论、让它从代码从零看一遍**，看会不会独立撞上同一问题。前者测对错，后者测独立命中数。

**派发**：每条经上面降误报流程存活的 P0/P1，另派**一个独立 reviewer**（新鲜眼、用对应维度 prompt、**不带原发现结论**），只让它重新审视该发现所在的代码位置，判断是否独立命中同一问题。插件形态按"P0/P1 × 独立复核"并行派发对应维度子代理。

**聚合**：
- **≥2 个独立 reviewer 命中同一发现**（原 reviewer + 交叉确认 reviewer）→ 标 `✓✓交叉确认`，阻塞清单可直接采信。
- **仅 1 个命中**（交叉确认 reviewer 未复现，且未给出 file:line 级反驳）→ 标 `⚠单源` 保留在阻塞清单，附"仅单一 reviewer 命中、建议人工复核"（可能是单 reviewer 的过度解读）。
- 若交叉确认 reviewer 给出 file:line 级**反证**（不是"没看到"而是"证明它不成立"）→ 并入上面的降误报流程当一张反驳票处理，不单独作数。

**双向增益**：为确认而派的独立 reviewer 重看这片代码时，常**附带捞回该处的新发现**——新发现按其严重度定级、照走本文件的对抗验证流程（新发现若为 P0/P1 也进交叉确认）。

**诚实边界**：插件子代理 `model: inherit`，两个 reviewer 同底模，独立性有限——`✓✓交叉确认` 是**弱独立**（比单源强，但不是真异构）。真正的强独立要靠**主会话换模型跑多轮对照**或**人工复核**（见 SKILL.md `## 使用建议：异构复审（强独立召回）`）。报告不得把 `✓✓交叉确认` 表述成"已被独立验证无误"。

## 输出补充（交叉确认结果，并入 consolidate）

```
### P0/P1 交叉确认结果
- [✓✓交叉确认] <P0/P1 发现原文> — 独立命中 2（原 reviewer + 交叉）
- [⚠单源] <P0/P1 发现原文> — 独立命中 1，交叉 reviewer 未复现（建议人工复核）
- [交叉附带新发现] <新发现原文> — 来源：交叉确认时捞回 — 待走对抗验证
```
