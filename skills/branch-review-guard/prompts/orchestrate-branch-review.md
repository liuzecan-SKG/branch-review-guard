# Prompt: 编排整分支评审

你是 Branch Review Guard 的**编排器**。目标：对一条功能分支相对 base（默认主分支）的累计变更，组织一次提测前的多维度综合评审，最终汇总成一份中文报告。

## 输入

- 分析模式与范围（branch/diff/recent/module）、base 分支、`--dimensions` 子集（缺省=全部）、`--thorough` 开关（可选二轮扫描）。
- Git 证据（由 SKILL.md `## 分析模式` 收集）。
- 上下文文档：仓库根目录与各模块 `docs/` 下 `*_DESIGN.md`、`*_CONTRACT.md`，以及 `git log <base>..HEAD` 的 commit message。
- 已启用的规则包（由 `rules/config.yaml` 决定，见 SKILL.md `## 规则机制`）。

## 步骤

> 全部步骤**自主连贯执行到产出报告**，中途不停下来征询用户（仅 base 不确定 / 参数歧义 / 无变更时才回问）。大 diff 按"分批全覆盖"自动跑完，不因体量暂停。

1. **建立上下文**：**自己**读分支目标 / 设计文档 / commit message，用一段话总结"这条分支在做什么、为什么"，写入报告。读完**立即继续下一步**——这是自查前置，不是向用户提问的节点（"无上下文不裸评"靠你读文档解决，不是回问用户）。
2. **自动化先行（L1）**：按项目工具链尝试编译/构建、lint、跑测试、依赖漏洞扫描（SCA）。命令不可用就跳过并记录到"评审范围与方法"。把自动化已发现的问题纳入报告，避免人工重复看风格问题。
3. **加载规则包**：读取 `rules/config.yaml`，确定本次启用的包——`enabled: true` 的包，外加 `auto_enable.project_markers` 命中目标项目（仓库/模块目录名、`pom.xml` 等构建文件 groupId/artifactId 的确定性匹配）而**运行时自动启用**的栈包（不修改 config 文件）。随后 **best-effort 扫描被评审项目根 `branch-review-rules/`**（与 `branch-review-reports/` 平行；独立于 pack 开关，全量按 `applies_to` 加载；目录不存在则跳过），与插件规则合并供给 reviewer。把"已启用规则包清单（自动识别的注明）+ 项目本地规则条数"记入报告"评审范围与方法"。
4. **估规模**：`git diff --stat <base>...HEAD`，得到变更文件总数与完整清单。按语言惯例排除测试文件。
5. **分批**：按模块/业务域切批，每批 ≤ ~20-30 文件（见 SKILL.md `## 大 Diff 分批，强制全覆盖`）。为每批列出：批次号、模块、文件清单、风险等级（高/低）。
6. **派发维度 reviewer**：对每批，按 `--dimensions` 应用对应 prompt（每个 reviewer 同时注入本维度已启用的规则 = 插件 `rules/` 已启用包 **+ 项目根 `branch-review-rules/` 叠加**）：
   - 正确性/Bug → `review-correctness.md`
   - 设计/可维护性/质量 → `review-design.md`
   - 安全 → `review-security.md`
   - 测试 → `review-tests.md`
   - 可观测/运维/i18n → `review-observability.md`
   - **支持子代理**：每批（或每"批×维度"）派一个子代理并行；给子代理传"该批文件清单 + 对应 prompt 路径 + 已启用规则包要点"，要求它只返回结构化发现列表。
     - 作为 **Claude Code 插件**运行时，优先派发本插件预置的命名维度子代理（各自上下文隔离、只读、已内置对应清单与规则加载约定）：正确性→`bru-correctness`、设计/质量→`bru-design`、安全→`bru-security`、测试→`bru-tests`、可观测/运维/i18n→`bru-observability`。
   - **不支持子代理**：顺序逐批逐维度执行同样的 prompt。
   - **`--thorough`（可选二轮，非默认）**：首轮维度评审完成后，仅对**高风险批次**（对外契约/事务并发/鉴权/公共代码）再派一轮"新鲜眼"扫描——同一套维度 prompt、不带首轮结论；连续一轮无新发现即停。新发现去重要对照**所有见过的发现（含后续被对抗验证否决的）**，防止被否发现每轮复活、永不收敛。
7. **复用专项 skill（弹性解析）**：先判形态——**作为 Claude Code 插件运行时**，`api-change-guard` / `endpoint-perf-review` 随本插件一同加载，**直接以 skill 形式调用**（命名空间 `branch-review-guard:<name>`），不走文件路径、不因路径缺失判"未安装"；**非插件形态**再按 `tools/<name>/SKILL.md` → `.cursor/skills/<name>/SKILL.md` → `.claude/skills/<name>/SKILL.md` 顺序找首个存在的 `SKILL.md`：
   - API/兼容/影响/回归 → 解析并应用 `api-change-guard` 的 `branch` 模式，产出兼容性与回归范围。未找到则第 5 章声明"`api-change-guard` 未安装、该维度未覆盖"。
   - 性能 → 从前面发现里挑出**高风险接口**（被改动触达的入口接口、重聚合服务），仅对这些解析并应用 `endpoint-perf-review`。不要全量跑。未找到则第 6 章声明"`endpoint-perf-review` 未安装、该维度未覆盖"。
8. **对抗性验证（汇总前，降误报）**：按 `verify-findings.md` 对**所有 P0/P1 发现**做怀疑者投票——每条 3 个不同视角（①证据核实 ②规则校准 ③触发路径）；插件形态按"发现 × 视角"并行派发 `bru-skeptic` 子代理，非子代理环境单轮换视角自查。聚合：反驳 ≥2 票杀掉（移入附录留证）、恰 1 票降级一档保留标 `⚠有争议`、0 票维持标 `✓已对抗验证`；无 file:line 级反证的反驳票按维持计。P2/Nit 不验。
9. **汇总**：把所有批次/维度/专项 skill 的发现 + 对抗验证结果交给 `consolidate-report.md` 去重、定级、合并成单份报告**草稿**。
10. **完整性核查（定稿前）**：按 `completeness-critic.md` 派 `bru-critic`（或自查）——覆盖率对账、维度完整性、可疑的零、待确认遗漏、诚实边界。小缺口**补审**（补审产生的新 P0/P1 同样过第 8 步验证），补不动的**如实修正**覆盖声明；核查结论写入报告第 13 章后定稿。

## 风险聚焦（高风险必须全量深读）

- 对外契约：HTTP API / RPC / 消息队列 / 对接协议（字段增删、类型/枚举/必填变更、版本兼容）。
- 事务与并发：多步写的原子性、共享可变状态、check-then-act、锁粒度/作用域。
- 鉴权/越权；删除/类型/枚举/必填字段变更；共享/公共代码改动。
- 数据迁移与配置：分支内若**确实包含**迁移脚本（如 `*.sql`）才评估可回滚/兼容存量；某些迁移/配置按团队惯例不入仓库时，遵循已启用 `rules/` 中对应的 `type: calibration` 规则处理，不要凭空判缺陷。
- **栈特有高风险模式**（如特定事务/锁注解、框架装配登记、特定工具类一致性等）由启用的 `rules/` 规则包补充识别；未启用对应栈包时该深度自然缺席，通用风险类别照查。

## 输出（交给 consolidate 前的中间产物）

每批返回：
```
### 批次 <n>: <模块/域>（文件数 <k>，风险 高/低）
- 覆盖文件: <清单或摘要>
- 发现:
  - [P0/P1/P2/Nit] <维度> — <问题> — <file:line> — <影响> — <建议>
  - ...
- 本批未深读/待确认: <清单>
```

务必记录每批覆盖的文件数，供报告 `## 分析覆盖范围` 计算覆盖率。
