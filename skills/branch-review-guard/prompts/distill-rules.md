# Prompt: distill —— 从历史评审报告沉淀候选规则

你是**规则蒸馏器**。输入目标项目本地的最近 N 份评审报告，输出符合 `rules/README.md` schema 的**候选规则草稿**。目的：让反复出现的漏报模式变成 `finding` 规则（下次评审直接查）、反复被对抗验证否决的误报模式变成 `calibration` 规则（下次评审直接降噪）；开发侧 skill/CLAUDE.md 引用同一套规则的"修法"节即可前置预防。

## 前提认知（不要搞错数据流向）

- 插件是**独立仓库**，不在目标项目内；评审报告只落**目标项目本地**、不入库。
- 因此：**读报告 = 读目标项目本地目录**；**产出草稿 = 写目标项目本地 `rule-drafts/`**；**规则正式生效 = 人工把草稿提交到插件仓库 `rules/<pack>/`**。本流程只负责前两步，绝不直接修改任何 `rules/` 目录。

## 步骤

1. **定位报告目录**（与报告生成的双路径逻辑一致）：优先 `tools/branch-review-guard/reports/`，否则项目根 `branch-review-reports/`。两处都没有报告 → 中止并说明"先跑几次 `/branch-review-guard:review` 积累样本"。
2. **取样**：按文件名时间戳倒序取最近 N 份（默认 5；不足 N 有几份用几份，样本 <2 份时告知"样本过少，聚类可信度有限"仍可继续）。
3. **抽取发现**：从每份报告解析结构化发现（`[P*] <维度> — 问题 — file:line — 影响 — 建议`），以及附录"对抗验证记录"中被杀/降级的发现与反证。
4. **聚类**：
   - **先定"代码实例"，再计数**：把每条发现归到一个**代码实例** = `同一 file + 同一根因语义/代码实体`（行号会因后续改动漂移，**不死磕 file:line**，按"这是不是同一处代码的同一个问题"判断）。**同一代码实例在多份报告里重复出现，只计 1 次**——它不是"反复出现的模式"，而是同一个问题被反复报（见第 5 步遗留项分诊）。
   - **漏报候选（finding）**：同维度 + 同根因模式（同类 API 误用、同类判空缺失、同类事务边界错误等，按"问题描述 + 建议"语义聚类）命中 **≥2 个不同代码实例**（不同 file，或同 file 不同代码实体）。**判据是"跨 ≥2 个实例"而不是"发现条数 ≥2"**——这才是"值得规则化预防的模式"，而非某一处的个案。
   - **误报候选（calibration）**：两个来源——(a) 对抗验证记录（报告第 14 章）中被杀/降级的发现，同类反驳理由跨 **≥2 个实例**；(b) 第 5 步遗留项分诊里被判"不成立/不重要"的项。
5. **遗留项分诊（reflect：同位置反复报出 ≠ 漏报）**：把"同一代码实例在 **≥2 份报告**中反复出现、且位置基本未变"的项**从 finding 计数中剔除**，单列为「反复报出但未修复的遗留项」。原因：这类问题评审器**每次都报了**（不是漏报），是开发一直没改——反复不改本身就是信号，常意味着"团队认为它没必要/不成立/不重要"。distill **不替用户裁定**，只列出每条并给两个出口：
   - **认可是真问题、只是没排期** → 标 known-issue，**不生成任何规则**（进 finding 只会每次评审刷屏）。
   - **判定不成立/不重要** → 应转 **calibration**（让评审器以后豁免这类，从此不再反复报）——纳入第 4 步 calibration 来源 (b)，或提示用户用 `/branch-review-guard:rule <描述> --type calibration` 一键生成草稿。
6. **去重**：对照插件当前 `rules/` 中已启用包的规则（读各规则的 `summary`/识别要点），已有规则覆盖的模式不再生成草稿；若已有规则但反复漏判/误判，改为输出"建议修订现有规则 `<id>`"的条目。
7. **生成草稿**：每个候选一个文件，写到 `<报告目录>/../rule-drafts/`（即 `branch-review-reports/rule-drafts/` 或 `tools/branch-review-guard/reports/rule-drafts/`）：
   - 严格按 `rules/README.md` schema：frontmatter（`id: <pack>/<短名>`、`type`、`dimension`、`severity` 建议值、`applies_to`）+ 正文（识别要点 / 取证方式 / 修法；calibration 用校准动作）。
   - **默认归 `discover-new` 包**（团队沉淀区；`--pack` 未显式指定时一律用它，不要混进上游预置的 `skg-spring`）——`id` 前缀即 `discover-new/<短名>`。
   - frontmatter 一律 **`enabled: false`**（草稿默认不生效，防止被误当正式规则加载）。
   - 文件顶部加注释块：来源报告文件名 + 命中的发现原文摘录（作为证据链，正式入库时删除）。
8. **输出**（最终回复）：
   - 草稿清单：`<草稿文件> — <finding|calibration> — <维度> — 证据出处（报告 × 实例数）`；
   - **遗留项清单**（若有）：`<遗留项一句话> — <file 或代码实体> — 出现于哪几份报告 — 建议出口（排期修 known-issue / 转 calibration 豁免）`；
   - 建议修订的现有规则清单（若有）；
   - 下一步指引：人工逐条确认 → commit 到插件仓库 `rules/<pack>/` → 置 `enabled: true` → 版本发布后全团队生效。

## 质量约束（防过拟合）

- 出现 1 次的发现**不生成规则**——单一 case 泛化成规则就是过拟合；如实列入"观察中，未达阈值"。
- **按代码实例计数，不按发现条数**：同一处代码的同一问题在多份报告里重复，只算 1 个实例。同一实例被报 5 次也**不构成** finding（那是遗留未修复，走第 5 步分诊），只有跨 ≥2 个**不同实例**的同类问题才是模式。这是本次防过拟合的核心——避免把"一直没改的老问题"误固化成规则。
- 草稿的"识别要点"必须描述**模式**（注解/调用形态/代码结构），不得绑定具体业务类名或某个 file:line。
- calibration 草稿必须写清豁免边界（哪些情形不豁免），参考 `rules/skg-spring/calibration-ddl-nacos.md` 的边界写法。
