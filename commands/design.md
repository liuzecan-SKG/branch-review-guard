# /branch-review-guard:design

对一条需求做**动手写码前**的多视角方案设计对比——评审侧 `:review` 守出口，设计侧的 `:design` 守入口。产出「对比表 + 推荐方案 + 嫁接落选亮点的综合方案」单份中文设计报告，并另出精炼设计稿（认可后移入 `docs/` 供 `:review` 建立上下文时自动读取）。

参数：`$ARGUMENTS`
- 第一段自由文本 = **需求描述**（必填，按优先级：参数文本 > `--input` 文档 > 自动扫 `docs/` 下 `*_DESIGN.md`/`*_REQUIREMENT.md`/`*_CONTRACT.md` + `git log` 推断；全部落空才回问一次）。
- `--input` = 需求/PRD 文档路径。
- `--module` = 限定勘察的模块路径前缀（缩小探索面 + 成本闸门）。
- `--variants` = 视角数，缺省按档位（quick=2 / 默认=3 / thorough=5）。
- `--quick` / `--thorough` = 成本档位（默认标准档；`--quick` 降为 2 视角 + 编排器抽查；`--thorough` 升为 5 视角 + 深度质证）。

> **自主一气呵成**：被调用后连贯跑完 P0→P6 并**直接产出报告**，中途**不要停下来问用户**「是否继续 / 是否开始」。澄清需求分三层（详见 skill 的 `## 需求模糊处理`）：**L1** 开跑前（P1.5）把承重假设打印给用户看一眼、**不停**同一回合直接续跑（用户可 Esc 纠偏）；**L2** 仅当「承重 ∧ P1 勘察后仍不可推断 ∧ 猜错会让推荐方案整个反掉」（或目标行为本身缺失/自相矛盾）才**恰好一次**终止性集中回问（**不承诺「不回复则按默认继续」**——纯 Markdown 无超时续跑通道）；**L3** 其余细节不确定进报告开头「假设清单」事后纠偏。能推断的绝不问。
> **设计阶段隔离即产品**：方案代理互不可见；顺序降级模式下方案独立性弱化（与评审侧「结果一致只是更慢」口径不同，报告如实声明）。

## 执行方式

1. 调用本插件 `design-panel` skill（`branch-review-guard:design-panel`），严格按其 `SKILL.md` 工作流程执行：P0 定标（需求自检 + 规模降档）→ P1 建事实底座 → P2 并行独立成案 → P3 对抗质证 → P4 裁判裁决 → P5 诚实边界核查 → P6 产出报告与精炼设计稿。
2. **优先用专用子代理并行**：本会话支持子代理时按本插件提供的子代理派发，互相上下文隔离、各自只回传结构化结果：
   - 设计代理（按视角并行 N）→ `dsp-designer`（视角：最小改动 / 长期可维护 / 风险与回滚 [+ 性能与容量 / 复用与删减]）
   - 对抗质证（每方案 1 个）→ `dsp-skeptic`（三角度：事实性 / 完备性 / 代价；四件套护栏见 skill 的 `prompts/challenge-claims.md`）
   - 裁判（定稿前 1 个）→ `dsp-judge`（打分 + 嫁接 + 推翻票抽验 + 完整性核查）
   不支持子代理时，按 SKILL.md 顺序多轮执行并声明独立性弱化（结论仅供参考）。
3. 强制诚实边界：承重论断每条带 file:line；性能/并发/迁移类只给「需运行时验证项」、不下「已验证通过」；被驳论断支撑的优势不计分。

最终回复遵循 SKILL.md `## 回复约定`：先报告链接、结论先行、再完整正文，正文后固定附衔接文案——

> 是否需要我把推荐方案展开成文件级开发 todo 清单（按依赖顺序排列，每项标注涉及文件与验收点）？开发中可随时用 `/branch-review-guard:diff` 边写边查，提测前用 `/branch-review-guard:review` 整分支评审。若认可精炼设计稿（`<slug>_DESIGN.md`），建议移入 `docs/` 提交——后续评审建立上下文时会自动读到它。
